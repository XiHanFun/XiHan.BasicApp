## 1. 执行摘要

### 1.1 整体印象

这是一套**工程素养在线、纪律性较强**的全栈中后台系统：

- 后端采用清晰的 DDD 四层（Domain / Application / Infrastructure / HttpApi）+ 模块化骨架，框架层（Core / Web.Core）很薄、职责干净；应用层做了严格的 CQRS 物理拆分（写 `AppServices` / 读 `QueryServices`），AppService 普遍很薄、逐方法挂 `[PermissionAuthorize]` + `[UnitOfWork]`，业务下沉到 DomainService。
- 领域模型基础扎实：统一雪花 `long` 主键（应用层赋值）、`RowVersion` 乐观并发、完整审计四件套、`TenantId` 非空多租户约定清晰且有文档注释，租户/软删过滤由全局 AOP 统一注入。
- 前端工程化程度高：`@`/`~` 双别名分层、动态分包、严格 `tsconfig`（`strict` + `noUncheckedIndexedAccess`）、token 刷新队列、请求签名加密层。

**但是**，在这套规整骨架之下，存在若干**系统性裂缝**：安全机制「建好了但没接上」、契约「靠人脑对齐」、质量「没有任何自动化护栏」、重构「留下大量考古层」。这些问题的共性不是「写错了」，而是「做了一半」——基础设施齐备，却给人「已完整」的错觉，这恰恰是最危险的状态。

### 1.2 值得肯定的亮点

- 多租户隔离与软删除由全局 `QueryFilter` AOP 统一兜底，仓储/服务无感。
- 雪花 ID 在前端已正确用 `string` 镜像，规避了 JS 53 位精度丢失。
- 错误处理普遍带结构化日志，抽样未发现空 catch 吞异常。
- 匿名端点收口良好（仅 LoginConfig / Login / RefreshToken 三个）。
- 仓储接口在 Domain、实现在 Infrastructure，依赖倒置正确。

## 2. 分维度详细问题清单

### A. 后端分层与目录结构一致性

> 整体：DDD 四层 + 模块化骨架规整，框架层职责干净，子域划分在多数层已对齐。主要问题是 Application 层「四套服务概念并存」、散落空孤儿目录、命名前缀不统一、框架契约零落地——多轮重构的沉积层未清理。

#### [高] A1. 四类「服务」概念并存且边界不清
- **证据**：
  - `Application/AppServices/Auth/AuthAppService.cs:38` — `[DynamicApi]` 对外编排入口
  - `Application/Services/Auth/AuthTokenIssueService.cs:27` — 无 `App` 前缀、无 DynamicApi 的内部辅助服务
  - `Application/QueryServices/Auth/AuthContextQueryService.cs:24-27` — 直接注入仓储做只读查询（全仓 53 个 QueryService 注入仓储）
  - `Domain/DomainServices/Authorization/Implementations/*` — 36 个领域服务，29 个注入仓储
- **问题**：`AppServices`（对外编排）/`Services`（应用内辅助）/`QueryServices`（只读）/`DomainServices`（领域逻辑）四类并存，`QueryServices` 与 `DomainServices` 都直接注入仓储做数据访问，读写路径分裂；`Services` 与 `AppServices` 的区别仅靠隐式约定，新人难判断新服务该放哪。
- **建议**：明文固化四类职责契约并写入模块 README；评估 `Services` 是否可并入 `AppServices` 私有协作者或下沉，减少一类概念。

#### [高] A2. 框架契约 `IQueryService` 全仓零实现（孤儿契约）
- **证据**：`framework/XiHan.BasicApp.Core/Queries/IQueryService.cs` 定义了读端契约，但全仓 `grep IQueryService` 仅命中定义文件本身，0 个 QueryService 实现或引用它，各服务仅实现自定义的 `IXxxQueryService`。
- **问题**：框架精心定义的读端抽象成为孤儿契约，丧失统一分页/投影/规约的切入点，框架约束形同虚设。
- **建议**：要么让 QueryService 基类继承 `IQueryService` 落地，要么删除该契约，不留空壳。

#### [中] A3. 三个空孤儿目录残留
- **证据**：`Services/Application/`（0 文件，与 `Application/Services` 反向重名）、`Repositories/RoleHierarchies/`（0 文件，正规仓储在 Domain/Infrastructure）、`Jobs/`（0 文件，实际作业在 `Infrastructure/Tasks/Jobs/DynamicJobWorker.cs`）。
- **问题**：三个顶层空目录暗示曾有「绕过分层的旁路结构」，误导读者以为存在第二套仓储/服务/作业体系。
- **建议**：直接删除。

#### [中] A4. `Domain/Entities/Enums` 与 `Domain/Enums` 双枚举目录命名易混
- **证据**：`Domain/Entities/Enums/`（37 文件，实体专属枚举，`*.Enum.cs` 双扩展命名）vs `Domain/Enums/`（3 文件，跨域共享枚举，如 `EnableStatus.cs`）。
- **问题**：用途确有区分但目录名肉眼难判该放哪，文件命名风格也不一（`SysRole.Enum.cs` vs `EnableStatus.cs`）。
- **建议**：语义化重命名（如 `Domain/Enums` → `Domain/SharedEnums`）+ README 说明归属判定；统一枚举文件命名风格。

#### [中] A5. `Infrastructure/Logging` 全量用 `Rbac` 前缀，与模块 `Saas` 命名冲突
- **证据**：`Infrastructure/Logging/RbacAccessLogWriter.cs` 等 8 个文件全用 `Rbac*` 前缀；对比 `Infrastructure/Auth/SaasUserStore.cs`、`Infrastructure/Security/SaasPasswordHistoryStore.cs` 用 `Saas` 前缀；接口侧却是无前缀 `IAccessLogWriter`。
- **问题**：同层内 `Rbac`/`Saas` 两套前缀并存，前缀承载的「归属模块」语义被撕裂。
- **建议**：统一前缀（建议 `Saas`），或全部去前缀仅靠命名空间区分。

#### [中] A6. 模块根类名 `XiHanBasicAppRbacModule` 与项目名 `Saas` 不一致
- **证据**：`modules/XiHan.BasicApp.Saas/XiHanBasicAppRbacModule.cs` — `namespace XiHan.BasicApp.Saas; class XiHanBasicAppRbacModule`，与 `XiHanBasicAppCoreModule`/`XiHanBasicAppWebCoreModule`/`XiHanBasicAppWebHostModule` 的「项目名+Module」约定不符。
- **建议**：重命名为 `XiHanBasicAppSaasModule`。

#### [中] A7. `AppServices` 命名空间不统一：多数扁平、少数带子域段
- **证据**：多数 `namespace ...Application.AppServices;`（扁平，如 `AuthAppService.cs:32`）；例外 `AppServices/Metadata/*`、`AppServices/Monitoring/*` 带 `.Metadata`/`.Monitoring` 子域段。
- **建议**：统一策略，建议把 `Metadata`/`Monitoring` 两处改回扁平。

#### [中] A8. 子域目录在各层覆盖不对齐，存在缺口
- **证据**：`Mappers` 缺 Metadata/Monitoring/System/Workbench/Auth（12 vs Dtos 18 子域）；`DomainServices`/仓储缺 Metadata/Monitoring/Logging/System/Workbench；`Logging` 域有 QueryServices/Dtos 但无领域服务与仓储接口（数据写入散在 `Infrastructure/Logging` 的 Writer 里），归属割裂。
- **问题**：部分缺失属正常（纯查询域无需领域服务），但缺乏显式标注，难分「刻意不实现」还是「遗漏」。
- **建议**：维护子域×分层覆盖矩阵并标注刻意留空的原因；收口 Logging 域数据写入路径。

#### [低] A9. 个别 AppService 直连仓储绕过领域/查询服务
- **证据**：`Application/AppServices/Profile/ProfileAppService.cs:55,92-103` 直接注入 `IUserPreferenceRepository` 读写（全仓 AppService 唯一例外）。
- **建议**：下沉到对应 DomainService/QueryService，保持 AppService 纯编排。

#### [低] A10. `HttpApi` 下 Admin/OpenApi/Public 三目录全空
- **证据**：三目录各 0 文件，控制器由 `[DynamicApi]` 自动生成。
- **建议**：纯动态 API 则删除空目录；若是分组规划则用占位 README 说明。

#### [低] A11. `Infrastructure/Tasks/Jobs` 层级冗余
- **证据**：`Infrastructure/Tasks/Jobs/DynamicJobWorker.cs` — `Tasks` 下仅 `Jobs` 一个子目录，`Jobs` 下仅 1 文件，叠加顶层空 `Jobs` 目录。
- **建议**：扁平化为 `Infrastructure/Jobs/`。

### B. 领域模型与数据访问

> 整体：领域模型基础扎实（雪花主键、乐观并发、审计四件套、租户非空、全局 AOP 过滤）。主要风险是分表正确性隐患、查询绕过仓储且不强制时间范围、平台表无差别套租户基类、聚合行为应用不一致。

#### [高] B1. 分表实体物理表名 token 与分表粒度不匹配
- **证据**：`Domain/Entities/SysLoginLog.cs:52`、`SysAccessLog.cs:50`、`SysOpenApiLog.cs:53` — 表名模板 `"Xxx_{year}{month}{day}"`（8 位日粒度 token）配 `SplitTable(SplitType.Month)`（月粒度）；注释写 `{yyyyMMdd}` 与代码 token 也不自洽。
- **问题**：`SplitType.Month` 下表名带 `{day}`，SqlSugar 会用某一天填充，生成形如 `SysLoginLog_20260601` 看似日表实为月表，运维按表名直觉清理时极易误判粒度；若不同实例对 `{day}` 填充理解不一致，还可能同月分裂多表。
- **建议**：月分表表名统一改为 `_{year}{month}`，或明确约定 `{day}` 恒为 `01` 并写死注释。

#### [高] B2. 分表查询不强制时间范围，易全月表扫描
- **证据**：`Application/QueryServices/Logging/LoginLogQueryService.cs:67-71` 直接 `DbClient.Queryable<>().Where(predicate).SplitTable()`，而谓词中 `LoginTimeStart/End`（:115-119）均可选；`AccessLogQueryService.cs:117-127` 同样；详情查询 `Where(x => x.BasicId == id).SplitTable()`（:101-104）按主键查却无时间维度，框架逐月表扫描直到命中。
- **问题**：实体注释明确「查询必须带时间范围否则扫描所有月表」（`SysAccessLog.cs:28`），但查询服务未在入口强制。随月表增长，无时间条件的分页/详情查询会线性放大为 O(月表数) 次扫描——典型的随时间劣化性能炸弹。
- **建议**：分页 DTO 时间起止设为必填或服务层兜底默认窗口；详情查询要求传入时间定位单张分表；下沉到封装「必带 SplitTable + 时间范围」的只读仓储。

#### [中] B3. 分表路由字段（CreatedTime）与业务时间字段（AccessTime）不同列
- **证据**：`SysAccessLog.cs:196-198` `[SplitField]` 在 `CreatedTime`，但查询排序/过滤用 `AccessTime`（`AccessLogQueryService.cs:70,120-126`）；写入器 `RbacAccessLogWriter.cs:59-62,85` 中 `CreatedTime=now`、`AccessTime=now-elapsed`，二者不同值。
- **问题**：插入按 CreatedTime（请求结束）路由分表，查询按 AccessTime（请求开始）过滤。跨月边界的长耗时请求会出现「记录物理落 N+1 月表但 AccessTime 属 N 月」的错位，若查询按 AccessTime 限定 N 月 + SplitTable 只扫 N 月表则漏查。SysLoginLog 同理。
- **建议**：让路由字段与对外业务时间字段同列，或写入时强制 `CreatedTime == AccessTime`。

#### [中] B4. 平台级账本表无差别继承租户基类
- **证据**：`Domain/Entities/SysMigrationHistory.cs:44-48` — 迁移历史（平台/连接级全局事实）继承含 `TenantId` 的 `BasicAppCreationEntity`，并建 `IX_{table}_TeId_CrTi` 租户索引。
- **问题**：迁移台账不属于任何租户，强制套多租户基类会落恒为 0 的冗余 `TenantId` 列、被全局租户过滤误作用、`IX_TeId_*` 前导列区分度为 0 浪费索引。
- **建议**：为平台级实体提供非多租户基类，迁移历史改用之。

#### [中] B5. 聚合「领域行为」应用不一致：部分改状态不发领域事件
- **证据**：`Domain/Entities/Aggregates/SysTenant.Aggregate.cs:30-41` `ChangeStatus` 守卫幂等 + `AddLocalEvent(...)`，规范；`SysConstraintRule.Aggregate.cs:27-38` `Enable()/Disable()` 仅 setter 赋值，无守卫无事件。
- **问题**：同名 `*.Aggregate.cs` 分部类，有的发领域事件有的只是 setter 包装，调用方无法形成稳定预期；状态变更不发事件意味着缓存失效/审计/通知靠手工补，易漏。
- **建议**：统一约定——聚合行为方法必须含状态守卫且对有外部关注的迁移发事件；纯 setter 不应伪装成聚合行为。

#### [中] B6. 查询服务直接注入 `ISqlSugarClientResolver` 绕过仓储层
- **证据**：`LoginLogQueryService.cs:39,44-49`、`AccessLogQueryService.cs:39,49` 注入 client resolver 直接 `DbClient.Queryable<>()`；对比仓储基类已封装 `CreateQueryable()`/`CreateNoTenantQueryable()`/`CreateWithDeletedQueryable()` 受控入口。
- **问题**：租户/软删 AOP 仍生效（非漏洞），但绕过仓储后分表约束无处收口、失去受控逃逸口、与分层意图不一致。根因是日志类无独立仓储。
- **建议**：为分表日志补只读仓储，QueryService 依赖仓储而非裸 client。

#### [低] B7. 主文件 + Expand 分部类拆分价值有限
- **证据**：`Domain/Entities/Expands/SysLoginLog.Expand.cs:22-41` 的 Expand 仅放两个 `[Navigate]` 导航属性（均 `IsIgnore=true`），一个实体散成 主体/Expand/Aggregate/Enum 最多四文件。
- **问题**：导航属性本不落库，剥到 Expand 收益不大，定位字段需跨文件跳转。属风格权衡非缺陷。
- **建议**：维持现状即可；若嫌跳转累可将导航属性并回主文件 region。

#### [低] B8. 日志冗余快照字段缺一致性说明
- **证据**：`SysLoginLog.cs:75-130`、`SysAccessLog.cs:70-161` 冗余 UserName/Browser/Os/Location，由 `RbacAccessLogWriter.cs:79-82` 写入时定格。
- **问题**：反范式快照本身正确，但属隐式约定，注释未声明「写入时快照不随主数据更新」，易被误加 JOIN 回填或误判脏数据。
- **建议**：实体注释补一句固化口径。

#### [低] B9. ValueObjects / Specifications 未落地（半套 DDD）
- **证据**：`Domain/{ValueObjects,Specifications}` 为空，仅 `Aggregates` 有 2 文件；框架层 `SqlSugarReadOnlyRepository.cs:25` 引用了规约能力但 Saas 模块未用。
- **问题**：DDD 战术模式只落地一小部分，基础设施齐备却几乎没用上（关联 R1），易让后续开发者误以为有完整体系可依赖。
- **建议**：明确技术取向——承认以「事务脚本+服务层」为主并写入文档，或补齐高价值场景规约。

#### [低] B10. 枚举跨实体复用约定未文档化
- **证据**：`Domain/Entities/Enums/*.Enum.cs` 与单一实体强绑定，若被其它实体复用会形成「实体 A 依赖实体 B 的枚举文件」隐式耦合。
- **建议**：约定「某枚举一旦被第二个实体引用即上移至 `Domain/Enums` 共享区」。

### C. 应用层、契约与 API 安全

> 整体：CQRS 物理拆分纪律强，三件套（Contracts/Dtos/Mappers）对齐度高，AppService 薄、逐方法挂权限与事务。主要风险集中在 **ABAC/FLS/数据范围「建模完整但读路径未执行」**，以及若干**按 id 操作未做业务归属校验**的潜在越权点。

#### [高] C1. 字段级安全（FLS）已建模但读路径完全未执行 → 见 H4
- **证据**：`Application/Mappers/Identity/UserApplicationMapper.cs:104-130` — `ToDetailDto`/`ToListItemDto` 直接平铺实体字段（含 Email/Phone/LastLoginIp），无脱敏/裁剪入参；全局检索 FLS 相关 19 文件全是 FLS 配置自身的 CRUD/缓存/事件，**无任何业务 QueryService/Mapper 在返回前调用 FLS 做读裁剪**；`EventHandlers/SaasDomainEventHandler.cs:116-138` 仅对 FLS 变更写审计。
- **问题**：FLS 配置可建可审计，但读真实数据时字段不脱敏。`IsReadable=false`/`MaskStrategy` 形同虚设，敏感字段对任何有 Read 权限者全量返回。
- **建议**：读路径统一插入 FLS 投影层（Mapper 之后增加 `IFieldLevelSecurityApplier` 后置处理或序列化阶段拦截）。

#### [高] C2. 数据范围（DataScope）未转化为行级查询过滤 → 见 H5
- **证据**：`QueryServices/Identity/UserQueryService.cs:96-133` `GetUserPageAsync`、`QueryServices/Authorization/RoleQueryService.cs:61-70` 列表查询筛选条件只来自入参 DTO，无「按当前登录者 DataScope 收敛可见行」逻辑；检索行级裁剪消费点仅命中读取配置本身。
- **问题**：角色配了 `DataPermissionScope`（SelfOnly/部门/全部），但分页查询不据此限制可见行——拥有 `User.Read` 的部门管理员能看到全租户用户。数据权限只在「能否进功能」生效，未落到「能看哪些行」。
- **建议**：引入统一数据范围解析器，QueryService 构造查询时按主体有效 DataScope 注入行级过滤，详情查询做归属校验。

#### [中] C3. 文件下载 / 预签名 URL 仅校验功能权限，缺业务归属校验（IDOR）
- **证据**：`AppServices/Files/FileAppService.cs:103-110` `DownloadFileAsync(long fileId)` 仅 `[PermissionAuthorize(File.Read)]` 后直接返回流；`:131-138` `GenerateFilePresignedUrlAsync` 按裸 id 生成可外发 URL，均无归属校验。
- **问题**：同租户内任何持 `File.Read` 者可遍历 `fileId` 下载/外链他人文件；预签名 URL 还把访问能力外泄到无鉴权链路。横向越权（IDOR）。
- **建议**：下载/预签名前校验文件归属，预签名有效期钳制上限。

#### [中] C4. 按 id 详情/删除依赖框架隐式租户过滤，应用层无显式兜底
- **证据**：`UserQueryService.cs:142-153`、`RoleQueryService.cs:79-90` 详情直接 `GetByIdAsync(id)` 无租户判断；`SaasDomainEventHandler.cs:561/599` 存在直接 `DbClient.Insertable().SplitTable()` 的写路径。
- **问题**：多租户安全建立在「框架过滤器永远生效」的隐式前提上，一旦某路径绕过过滤器即可能跨租户读，应用层无防线（关联 R2）。
- **建议**：关键聚合的详情/删除增加显式归属断言，补「租户 A 用 id 访问租户 B 应 404」集成测试。

#### [中] C5. `GetByUserNameAsync` 登录定位仅按 UserName，未带 TenantId
- **证据**：`Infrastructure/Repositories/Identity/UserRepository.cs:31-39` — `Where(u => u.UserName == userName).FirstAsync()` 无 TenantId 谓词；而 `SysUser.cs:65` 唯一约束是 `UX_TeId_UsNa`（TenantId+UserName 才唯一）。
- **问题**：登录定位依赖调用前是否已建立正确租户上下文，若上下文为空/过滤未生效，同名跨租户用户 `FirstAsync` 取值不确定，存在登录串号风险。仓储语义与唯一索引契约不一致。
- **建议**：方法增加显式 `tenantId` 入参参与查询，与 `UX_TeId_UsNa` 对齐。

#### [中] C6. 应用层无声明式入参校验，DTO 全裸属性
- **证据**：`Application/Dtos` 下 `[Required]/[MaxLength]/AbstractValidator` 0 命中；`RoleCreateDto.cs:23-69`、`AuthDtos.cs:57-88` 均纯属性；校验靠手写守卫 `AppServices/SaasCommandValidation.cs` 分散在应用/领域层。
- **问题**：缺统一声明式校验，长度/必填/范围约束易漏（如 `MaxMembers/Sort` 无下限），前端无法从契约/Swagger 推断约束。
- **建议**：入参 DTO 补 DataAnnotations 或 FluentValidation 管线，形成「声明式拦边界 + 领域不变量兜底」双层，并让校验进入 OpenAPI 契约。

#### [中] C7. 无 OpenApi/Public 分层，外部接口签名校验与限流缺位
- **证据**：`HttpApi/Admin|OpenApi|Public` 目录不存在控制器，全部经 `[DynamicApi(Group="BasicApp.Saas")]` + 基类 `[Authorize]` 暴露；检索签名校验/限流在 Application 下无任何机制。
- **问题**：所有接口都是内部后台型鉴权（JWT+权限码），未区分对外开放 API；若未来暴露 OpenApi，缺 AK/SK 签名、时间戳防重放、租户级限流落点。
- **建议**：有对外规划则在框架/中间件层补签名+限流并按分组区分策略；暂无则文档明确「全部内部接口」。

#### [低] C8. 登录主流程与审计/会话写入同 UoW 事务
- **证据**：`AuthAppService.cs:138-232` `LoginAsync` 标 `[UnitOfWork(true)]`，事务内签发 token、写会话后 `await _localEventBus.PublishAsync(...)`；处理器 `SaasDomainEventHandler.cs:31` 同 UoW 内 `DbClient.Insertable().SplitTable()` 写审计。
- **问题**：LocalEventBus 同步分发，审计日志写失败会回滚整个登录事务，「登录被一条日志拖垮」。审计应是旁路最终一致。
- **建议**：审计/差异日志事件改为异常吞掉并记录或独立事务/异步队列，与核心写分离。

#### [低] C9. 手写 Mapper 全静态逐字段，存在新增字段漏映射风险
- **证据**：`UserApplicationMapper.cs` 全手写静态逐字段（ListItem 18 字段、Detail 19 字段、CreateCommand 20 参数），全模块 50 个同构 Mapper，无 Mapster/AutoMapper，无编译期校验。
- **问题**：实体新增字段时无任何编译/测试提醒，易「实体加了字段但 DTO 漏映射」。优点是显式无反射开销。
- **建议**：保持手写可接受，但对关键聚合补「字段覆盖快照测试」或引入 Mapster 源生成器。

### D. 前端架构

> 整体：工程化程度高于同类模板（依赖倒置、动态分包、严格 tsconfig、token 刷新队列、签名加密层）。但存在系统性裂缝：`packages` 反向依赖 `src` 打破分层契约；schema 引擎止步于「列表+搜索」却制造大量类型断言；vue-i18n 基建几乎未被使用；客户端 bundle 硬编码签名/加密密钥。

#### [高] D1. `packages` 反向依赖 `src`，分层契约被打破 → 见 H8
- **证据**：`packages/components/schema/types.ts:3` `import { ApiId, PageResult } from '@/api'`；`useSchemaTable.ts:2` 同；`packages/composables/useEnum.ts:3` `import { enumMetadataApi } from '@/api'`（运行时值导入）；共 8 个 `packages` 文件 `from '@/...'`。
- **问题**：`app-context.ts:1-22` 明确契约「packages 只用 `~/` 引用自身，src 用 `@/` 注入」，但 schema 核心/composables/layouts 直接 import `@/api`，使 `packages` 无法脱离 `src` 独立复用，「可复用基建」名不副实（关联 R5）。
- **建议**：契约类型下沉到 `packages/types` 由 `src/api` 反向引用；运行时依赖通过 `app-context.apis` 注入；加 ESLint `no-restricted-imports` 禁止 `packages/**` 出现 `@/`。

#### [高] D2. 客户端 bundle 硬编码签名/加密密钥 → 见 H1
- **证据**：`.env.production:10-11` `VITE_API_SECURITY_SECRET_KEY=basicapp_frontend_signature_secret_2026`、`VITE_API_SECURITY_ENCRYPT_KEY=...`；`security.ts:48-50` 读取并参与 HMAC 签名/AES 加密。
- **问题**：`VITE_` 变量在 build 时内联进前端产物，任何用户可从 JS 读出 HMAC/AES 密钥。若后端真依赖该签名防篡改，则攻击者零成本绕过——是「安全感」而非安全。
- **建议**：明确该前端签名仅用于轻量防爬，真实机密性靠 HTTPS+服务端鉴权；生产密钥从仓库移除改 CI 注入，文档写明它是公开值。

#### [高] D3. token 明文存 localStorage，易受 XSS 窃取 → 见 H9
- **证据**：`stores/access.ts:8-9` `accessToken/refreshToken = LocalStorage.get(...)`；`request/index.ts:131-133` 每次请求取 token 注入；`pinia-plugin-persistedstate`（package.json:74）已装但全 `packages` 无 `persist` 使用，持久化全靠手写 LocalStorage。
- **问题**：access/refresh token 都明文存 localStorage，任意 XSS 可读取并长期冒用（refresh token 尤危）；同时存在「手写 LocalStorage」与「已装未用的 persistedstate」两套机制，后者是死依赖。
- **建议**：refresh token 优先走 HttpOnly Cookie；若必须前端存则集中到单一持久化层并明确风险；清理或统一持久化插件。

#### [高] D4. Schema 引擎抽象不彻底，催生大量类型断言 → 见 H10
- **证据**：`SchemaPage.vue:24` `type Row = Record<string, any>`；`src/views/system/role/index.vue` 约 15 处 `(row as unknown as RoleListItemDto)`，`resource.page` 返回 `as unknown as Promise<PageResult<Record<string, unknown>>>`（:178）；`selectors.ts:116` `column as unknown as DataTableColumn<TRow>`。
- **问题**：schema 用 `Record<string, any>` 抹平行类型绕开 Vue 泛型协变限制，代价是把类型安全甩回每页——页面反复 `as unknown as`，字段改名编译器不报错（关联 R1）。
- **建议**：用泛型工厂 `defineSchema<TRow>()` 让回调在定义点携带真实类型，框架内部做一次类型擦除，把断言收敛到框架 1 处而非页面 N 处。

#### [中] D5. Schema 引擎只覆盖「列表+搜索」，CRUD 仍每页手写
- **证据**：`types.ts:181-191` `SchemaResource` 只有 `page/tree/remove`，无 create/update/detail；`role/index.vue` 共 668 行，schema 约 130 行，其余 ~530 行是手写表单/详情/onAction 分发；`ListFieldSchema` 声明的 `editable/required/readonly/importable/exportable`（types.ts:58-62）引擎未消费。
- **问题**：字段 schema 标榜「搜索/列/导出/导入/详情均由此派生，禁止重复定义」，实际只派生列与搜索，表单/详情/导入导出全手搓，字段定义事实上重复，违背自身单一事实源承诺；引擎复杂度（8 文件、3 套 settings 状态机 + viewManager）与省下的代码不成正比。
- **建议**：补齐 `SchemaFormPanel`/`SchemaDetailPanel` 兑现单一事实源，或收缩 schema 范围删掉未用字段。当前「半成品抽象」是最差状态。

#### [中] D6. vue-i18n 基建齐全但应用层几乎不用，硬编码中文遍地
- **证据**：`packages/locales/langs/{zh-CN,en-US}` 词条完整；全 `src/views` 仅 6 文件用 `t(`/`useI18n`，其余 50+ 视图零 i18n；`role/index.vue` 标题/按钮/枚举标签全硬编码；连引擎 `SchemaPage.vue:135/166/250/269` 都硬编码「操作/更多/刷新/全屏」。
- **问题**：维护双语词条基建却在业务层完全旁路，既无法真正切语言（覆盖率极低）又白白承担维护负担（关联 R1）。
- **建议**：二选一——不做国际化则删 en-US 与 vue-i18n 依赖；要做则 schema 层强制 title 走 i18n key + lint 拦截模板中文字面量。

#### [中] D7. 前端权限校验逻辑分裂为两套数据源
- **证据**：`usePermission.ts:11` `userStore.hasPermission(p) || accessStore.hasCode(p)`，权限码同存 user 和 access 两 store；`user.ts:32` 与 `access.ts:51` 各实现一份判断；路由守卫 `guard.ts:152-153` 只查 `userStore`，与 `usePermission` 口径不一致。
- **问题**：同一权限码写入两 store 再 OR，任一处更新不同步会「按钮能点但路由进不去」或反之。
- **建议**：权限码收敛到单一 store，`usePermission` 与路由守卫共用同一判定函数。

#### [中] D8. 重型编辑器组件未懒加载且为死代码
- **证据**：`components/common/index.ts:1-4` 静态 export RichTextEditor/MdEditor/JsonEditor（tiptap/md-editor-v3/jsoneditor），无 `defineAsyncComponent`；这些组件全仓仅被 `packages/views/_core/editor-demo/index.vue` 一个 demo 页引用；`vite.config.ts:93-109` 已为其单独分包说明体积可观。
- **问题**：重组件通过 barrel 静态导出，削弱 tree-shaking，且实际只服务一个 demo 页，近乎死代码却仍占构建与维护成本（关联 F7）。
- **建议**：改 `defineAsyncComponent` 并移出公共 barrel；demo 若仅示例考虑移出生产构建。

#### [低] D9. `resolveUrl` 硬编码 `/api/` 前缀 — ✅ 已修复

#### [低] D10. 列表查询契约三层映射，role 页漏传排序

### E. 前后端契约一致性

> 整体：契约对接质量中上，前端 DTO/枚举为纯手写镜像（非 codegen），作者明显理解后端序列化规则（雪花 ID 已用 string、审计枚举用字符串镜像）。核心风险是手写镜像缺乏机器校验，以及非 ID 的 `long` 字段类型错配。

#### [高] E1. 非 ID 的 `long`（耗时）后端序列化为字符串，前端类型标 `number`（类型谎言）→ 见 H11
- **证据**：后端 `XiHan.Framework/.../LongJsonConverter.cs:39` `writer.WriteStringValue(value.ToString())` 把所有 `long` 序列化为字符串；`AccessLogListItemDto.cs:113`/`OperationLogListItemDto.cs:84` `long ExecutionTime`；前端 `access-log.types.ts:39`/`operation-log.types.ts:42` `executionTime: number`。
- **问题**：后端实际下发 `"executionTime": "123"`（字符串），前端 TS 声明 `number`，运行时拿到字符串，数值比较/排序产生字典序错误（`"100" > "99"` 为 false）。ID 字段已用 `ApiId=string` 规避，非 ID 业务 long 漏网。
- **建议**：所有后端 `long` 的非纯展示字段前端统一标 `string`（同 `ApiId` 策略），需运算处显式 `Number()`。根治见 E2 codegen。

#### [高] E2. 前端 DTO/枚举纯手写镜像，无 codegen，全靠人工对齐 → 见 H12
- **证据**：前端类型分布在 `src/api/modules/**/*.types.ts` 全手写，文件头注释「与后端 JsonStringEnumConverter 序列化值一致」证明人工对照；无 swagger/openapi 生成脚本痕迹；抽样三组日志 DTO 当前对齐但无机制保证后端改动后前端同步。
- **问题**：后端增删字段/改可空性/改枚举值前端不报错不感知，只能靠 review 人肉发现，高频迭代模块漂移风险随规模线性上升（关联 R3）。
- **建议**：引入 OpenAPI codegen（后端已是动态 API + Swagger）替换手写类型，或至少加「swagger.json → 前端类型快照」契约测试在 CI 比对漂移。

#### [中] E3. 统一返回 `code` 后端序列化为枚举字符串名，前端按数字比较（死分支）
- **证据**：后端 `ApiResponse.cs:28` `ApiResponseCodes Code`，枚举 `Success=200` 但**无 `[JsonConverter]` 数字覆盖**，全局注册 `JsonStringEnumConverter`（`XiHanAspNetCoreMvcOptions.cs:131`），故 `code` 实际下发 `"Success"`；前端 `common.ts:15` `code: number | string`，`request/index.ts:336` `responseCode === BIZ_CODE.SUCCESS(200) || === 0`。
- **问题**：`code` 实际是 `"Success"`，前端 `=== 200`/`=== 0` 永不命中，靠前面的 `isSuccess === true` 兜住未爆雷，但数字比较分支是事实死代码，一旦后端 `isSuccess` 缺失或前端改逻辑就误判。`BIZ_CODE` 数字常量与后端字符串枚举名语义对不上。
- **建议**：二选一定死契约——后端给 `Code` 加数字序列化，或前端 `BIZ_CODE` 改字符串枚举名。

#### [中] E4. 枚举四处重复定义，OperationType 后端还有两份
- **证据**：后端 `SysAccessLog.Enum.cs` `AccessResult{Success=0..}` 带 `[Description("成功")]`；前端 `access-log.types.ts:4-11` 字符串镜像枚举；前端 `views/log/access/index.vue:21-26` 又手写 `accessResultOptions` 把中文 label 抄一遍；`OperationType` 后端有两份（`SysOperation.Enum.cs:59` 与 `SysDiffLog.Enum.cs:59` 值相同）。
- **问题**：同一枚举的「值」与「中文标签」散在 4 处，改一处漏三处；后端两份 OperationType 哪份权威不明。
- **建议**：后端提供枚举元数据接口供前端动态拉取生成 options，或 codegen 同时产出枚举与 label；后端两份 OperationType 合并。

#### [中] E5. `DateTimeOffset` 序列化为无时区格式，丢失时区信息
- **证据**：后端 `JsonConverterFactory.cs:71` 注册 `DateTimeOffsetJsonConverter("yyyy-MM-dd HH:mm:ss", isUtc=false)`；前端 `access-log.types.ts:34` `accessTime: DateTimeString`（即 string）按裸字符串处理。
- **问题**：下发 `"2026-06-01 12:00:00"`——无 `T`、无时区偏移，非 ISO 8601。前端 `new Date()` 跨浏览器/时区解析不一致，跨时区用户时间错位，`DateTimeOffset` 时区语义被丢弃。
- **建议**：后端改 ISO 8601（`"o"` 或 `yyyy-MM-ddTHH:mm:ssK`）保留时区；前端用 dayjs 显式解析格式。

#### [低] E6. 分页查询用展平的 MVC 绑定查询串，请求/响应枚举双轨
- **证据**：前端 `access-log.types.ts:13` `AccessLogPageQueryDto extends PageRequest`（嵌套对象），但 `base.ts:85-135` 展平成 `Conditions.Filters[0].Operator`、`Page.PageIndex` 用 GET query string 传输；`QueryOperator`（types.ts:7）请求里走数字枚举，响应里同名枚举走字符串。
- **问题**：契约靠「MVC 查询串大小写不敏感绑定 + 数组下标语法」隐式成立，同一枚举请求数字/响应字符串两种线格式，认知负担高且无类型保护。
- **建议**：分页查询改 POST + JSON body 统一 camelCase + 字符串枚举，或对展平逻辑加单测固定契约。

#### [低] E7. 刷新令牌端点硬编码，权限码无类型约束
- **证据**：刷新端点 `request/index.ts:275` 硬编码 `'/Auth/RefreshToken'`，绕过 `src/api/modules` 客户端约定；权限码后端 `List<string> permissions`、前端 `auth.ts:16,91` `string[]`，前后端无共享常量。
- **问题**：后端调路由前端不会被覆盖；权限码裸字符串拼错静默失效。
- **建议**：刷新端点走统一客户端；后端导出权限码常量供前端生成联合类型。

#### [低] E8. 错误提示无前端错误码映射表，直接透传后端 message
- **证据**：`request/index.ts:351` 业务失败 `Promise.reject(new Error(responseMessage))` 直接用后端文案；全仓无 `errorCode→message` 映射；后端 `ApiResponse.cs:33` Message 用 `Description()` 中文。
- **问题**：前端完全依赖后端文案，无法 i18n，无法对特定业务码差异化交互，错误提示散落各 view catch。
- **建议**：建立前端 `code → 文案/动作` 映射层，结合 E3 把 code 契约定死后做集中错误处理。

### F. 工程化与质量保障

> 整体：分层规范、命名考究、`.gitignore`/props 构建配置组织良好，错误处理普遍带结构化日志。但存在三个系统性硬伤——全仓零测试、密钥明文入库、无 CI/CD，使代码质量缺乏任何自动化护栏。

#### [高] F1. 全仓零自动化测试，关键域无任何覆盖 → 见 H2
- **证据**：全仓 `*.Tests.csproj` = 0；`*.spec/*.test` 仅命中 node_modules；后端 989 个 `.cs` 源文件无对应测试工程。
- **问题**：鉴权（`AuthenticationDomainService`/`AuthTokenIssueService`/`PasswordPolicyDomainService`）、多租户、动态任务、查询条件构造（`QueryConditionsExtensions`）等高风险逻辑无回归保护，重构靠手测，缺陷易回流。本仓最大短板。
- **建议**：至少为鉴权与密码策略、`QueryConditionsExtensions`、SaaS 配置解析建 xUnit 单测；前端为 store/拦截器引入 Vitest。优先纯函数与领域服务。

#### [高] F2. 开发配置含全套明文密钥且已被 git 跟踪 → 见 H1
- **证据**：`appsettings.Development.json` JWT `SecretKey`（行 23）、DB `Password=postgres`（行 66）、Redis `password=redis`（行 114）、OpenApi `SecretKey/EncryptKey`（行 163-164）；`git ls-files` 确认**已跟踪**；两处 `.gitignore` 只忽略 `appsettings.Production.json`，不忽略 Development 与 `.env`；前端 `.env`/`.env.production` 同样含密钥且已跟踪。（本报告已亲自核验坐实）
- **问题**：JWT 签名密钥泄漏意味着任何拿到仓库者可伪造 token；开发密钥常被复制到生产或用于推断密钥规律；前后端密钥对都已入库。
- **建议**：敏感项移出版本库（User Secrets / 环境变量 / 占位符），`.gitignore` 忽略含密钥的本地配置，轮换已泄漏的 JWT/OpenApi/DB/Redis 密钥。

#### [高] F3. 无 CI/CD 流水线，无任何质量门禁 → 见 H3
- **证据**：`.github/workflows` 为空，无任何 `.yml/.yaml`。
- **问题**：build/lint/type-check/test 全靠本地手动；前端虽配 lint/type-check/build 脚本与 simple-git-hooks，但无服务端强制门禁，后端无任何自动化校验，PR 合入无卡口。
- **建议**：建 GitHub Actions——后端 `dotnet build`（+ 建测试后 `dotnet test`），前端 `pnpm lint && type-check && build`，设为 PR 必过检查。

#### [中] F4. CodeGeneration 模块名不副实，无生成器实现（空壳）
- **证据**：`modules/XiHan.BasicApp.CodeGeneration` 仅有 Domain/Entities（5 实体）、Enums、Seeders、Module 装配，**无任何模板引擎/生成器/代码输出逻辑**；README 自承提供代码生成能力但未实现；`WebHost.csproj:18` 引用了它。
- **问题**：以「CodeGeneration」命名却只有空壳领域模型+菜单种子，实际不生成任何代码，还向主机注册菜单/权限种子（增加无效菜单与运行期开销），属未完成或废弃功能桩（关联 R1/R5）。
- **建议**：明确 WIP 还是死代码——短期不实现则从 csproj 摘除引用、停注册种子；保留则补生成器并 README 标注「领域模型已就绪，生成器待实现」。

#### [中] F5. 业务异常消息全硬编码中文，国际化仅覆盖枚举
- **证据**：本地化资源仅 `Enums.en-US.json`/`Enums.zh-CN.json`；业务异常硬编码中文，如 `ProfileVerificationService.cs:60 "验证码无效或已过期。"`、`:76`、`CacheManagementService.cs:101 "缓存键不能为空"`、`TaskSchedulerSyncService.cs:76`。
- **问题**：配置了 `DefaultCulture: zh-CN` 与本地化框架却只资源化枚举，用户可见错误消息无法多语言（与后端/前端 i18n 半套问题呼应，关联 R1）。
- **建议**：用户可见业务异常走本地化资源键；开发者向的 `ArgumentException` 可保留硬编码但统一约定。

#### [中] F6. 前端无全局错误边界与错误上报
- **证据**：`src` 下 `app.config.errorHandler / window.onerror / onErrorCaptured / Sentry` 仅命中自动导入声明，无实现。
- **问题**：未注册 Vue 全局 errorHandler，无未捕获异常兜底与上报，生产前端崩溃无可观测性。
- **建议**：入口注册 `app.config.errorHandler` 与 `window.onerror/unhandledrejection`，接入轻量错误上报。

#### [中] F7. 前端并存三套编辑器，依赖偏重
- **证据**：`package.json` 同时引入 `@tiptap/*`（12 包，行 56-66）、`md-editor-v3`（行 71）、`vue3-ts-jsoneditor`（行 79）。
- **问题**：Tiptap 与 md-editor-v3 功能高度重叠（都做富文本/Markdown），同时打包显著增大产物（与 D8 死代码呼应）。
- **建议**：核对实际引用点，富文本场景二选一；JSON 编辑器按需懒加载。

#### [中] F8. 后端无中央包管理（Directory.Packages.props）
- **证据**：全仓无 `Directory.Packages.props`；`modules` 各 csproj `PackageReference` = 0（靠 framework 传递），版本由 `props/` 下 version.props 约束。
- **问题**：当前模块层无直接包引用暂未暴露版本分散，但一旦任一模块新增直接 NuGet 引用即有版本漂移风险。
- **建议**：引入 `Directory.Packages.props` 启用 CPM，把版本约束收口。

#### [低] F9. 设计/重构文档与实现可能不同步
- **证据**：`docs/refactor/` 下有 `00-总览/01-仓储层重构/02-服务层重构/02-附录-上帝服务拆分清单/05-UI美化重构` 等（均 2026-06-01 同一时间戳疑批量生成）。
- **问题**：大量「重构计划」类文档（含「上帝服务拆分清单」）说明存在已知技术债，但计划态还是已落地无法从时间戳判断，易与实现脱节。
- **建议**：文档头标注状态（计划/进行中/已完成）与对应实现位置，已落地的归档，未落地的转 issue。

#### [低] F10. CodeGeneration 模块残留 `.Backup.tmp`
- **证据**：`modules/XiHan.BasicApp.CodeGeneration/XiHan.BasicApp.CodeGeneration.csproj.Backup.tmp`（未被跟踪，`.gitignore` 的 `*.tmp` 已覆盖，本地残留）。
- **建议**：清理；低优先级卫生问题。

## 4. 方法论与边界说明

- **审查方式**：六个维度并行静态代码审查，所有 finding 均附实际读到的「文件:行」证据，未基于臆测。最高危的「密钥入库」已由审查者亲自 `git ls-files` + 内容核验坐实。
- **未覆盖范围**：`XiHan.Framework`（独立仓库，`backend/src/framework` 仅含 BasicApp 适配层）的全局过滤器、全局异常中间件、序列化器实现不在本仓，部分多租户/FLS 的最终执行行为需结合框架仓与集成测试进一步确认；本报告中相关越权判断建立在「框架过滤器若失效则应用层无兜底」这一可验证前提上。
- **前端抽样**：`src/views` 深读 `system/role` 一例作为 schema 落地样本，其余同构页面（user/permission/org/tenant/log/*）预计存在相同的 `as unknown as` 与硬编码中文模式。
- **结论定调**：本项目骨架与工程素养在同类项目中属中上水平，绝大多数问题不是「能力问题」而是「完成度问题」——把已经搭好的安全/契约/抽象/质量基础设施真正接通并加上自动化护栏，即可显著提升整体健壮性。
