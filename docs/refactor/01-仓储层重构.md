## 1. 现状（实证，非臆测）

### 1.1 抽象：空标记接口 + 两类基类
```csharp
// Domain/Repositories/Abstractions/ISaasRepository.cs
public interface ISaasRepository<TEntity> : IRepositoryBase<TEntity, long>
    where TEntity : class, IEntityBase<long> { }

// Domain/Repositories/Abstractions/ISaasAggregateRepository.cs
public interface ISaasAggregateRepository<TAggregateRoot> : IAggregateRootRepository<TAggregateRoot, long>
    where TAggregateRoot : class, IAggregateRoot<long>, new() { }
```
对应实现 `SaasRepository<T>`（普通实体）与 `SaasAggregateRepository<T>`（聚合根，注入 `IUnitOfWorkManager`）。

### 1.2 具体仓储已经很薄（`UserRepository`）
```csharp
public sealed class UserRepository(ISqlSugarClientResolver clientResolver, IUnitOfWorkManager uow)
    : SaasAggregateRepository<SysUser>(clientResolver, uow), IUserRepository
{
    public async Task<SysUser?> GetByUserNameAsync(string userName, CancellationToken ct = default)
        => await CreateQueryable().Where(u => u.UserName == userName).FirstAsync(ct);

    public async Task<bool> ExistsUserNameAsync(string userName, long? excludeUserId = null, CancellationToken ct = default)
    {
        var q = CreateQueryable().Where(u => u.UserName == userName);
        if (excludeUserId.HasValue) q = q.Where(u => u.BasicId != excludeUserId.Value);
        return await q.AnyAsync(ct);
    }
}
```
> 命名规范（`GetByXxx`/`ExistsXxx`）、可空语义、`CancellationToken` 一致，**无需改造**。

### 1.3 聚合区分清单（实测）
- 聚合根仓储（`ISaasAggregateRepository`）：User、Role、Permission、Operation、Resource、OAuthApp、Department、Tenant、Review、Task（10 个）。
- 普通实体仓储（`ISaasRepository`）：其余约 41 个（绑定关系类、日志类、配置项类等）。
- ⚠️ **已发现一处真实误分类**：`SysConstraintRule` 具备聚合语义（有 `SysConstraintRule.Aggregate.cs` 领域行为文件、含 `Items` 子实体），但 `IConstraintRuleRepository : ISaasRepository<SysConstraintRule>`（**普通仓储**）。这正是下文 R-E 审计要兜住的问题——应改用聚合仓储或明确其非聚合定位。

### 1.4 查询范式二元并存（**核心问题**）
分页/列表查询主流走动态 `QueryConditions`（`UserQueryService` 实证）：
```csharp
request.Conditions.SetKeyword(keyword.Trim(),
    nameof(SysUser.UserName), nameof(SysUser.RealName), nameof(SysUser.NickName),
    nameof(SysUser.Country), nameof(SysUser.Remark));
request.Conditions.AddFilter(nameof(SysUser.Status), status.Value);
request.Conditions.AddSort(nameof(SysUser.CreatedTime), SortDirection.Descending, 1);
var users = await _userRepository.GetPagedAsync(request, ct);   // 通用执行
```
而 `Domain/Specifications` 仅 8 个、且**应用层 0 处引用**：
```

Specifications/Authorization/ValidUserRoleSpecification.cs
Specifications/Identity/ActiveUserSpecification.cs
Specifications/Tenancy/AvailableTenantSpecification.cs  ……（共 8）
```

## 2. 真实问题与目标

| 编号 | 真实问题 | 目标 |
| --- | --- | --- |
| R-A | 查询范式两套并存、心智不统一 | 明确边界：**列表/分页查询用 QueryConditions；跨实体复杂业务判定用 Specification**，并写入规范 |
| R-B | `QueryConditions` 字段名用裸 `nameof` 字符串散落，重命名属性不报错 | 为每实体提供**字段引用常量/强类型 selector**，集中维护 |
| R-C | 8 个 Specification 仅领域层用，价值未释放 | 评估：要么扩大到所有复杂只读判定，要么收编为领域不变量校验，避免“半套规约” |
| R-D | 空标记接口未承载共性 | 可选：在 `ISaasRepository` 增补真正通用的便捷只读方法（如 `GetByIdsAsync`） |
| R-E | 聚合分类正确但缺审计基线 | 出一份“聚合根 ⇄ 仓储类型”对照表纳入文档，防后续新增实体走错基类 |

## 3. 设计与代码模板

### 3.1 字段引用类型安全化（R-B，优先级最高、收益最大）
保留 `QueryConditions` 范式（它已支撑 30 个查询服务、与前端动态参数天然契合），但**消除裸字符串字段名**。

方案：为每实体生成/手写一个字段名常量类，或提供基于表达式的扩展方法。

```csharp
// 方案一：字段名常量（轻量，零运行时开销）
// Application/QueryServices/Identity/UserQueryFields.cs
internal static class UserQueryFields
{
    public const string UserName = nameof(SysUser.UserName);
    public const string RealName = nameof(SysUser.RealName);
    public const string NickName = nameof(SysUser.NickName);
    public const string Status   = nameof(SysUser.Status);
    public const string CreatedTime = nameof(SysUser.CreatedTime);
}
```
```csharp
// 方案二（推荐）：表达式重载，编译期校验属性存在
public static QueryConditions AddFilter<T, TValue>(
    this QueryConditions c, Expression<Func<T, TValue>> selector, TValue value)
    => c.AddFilter(GetMemberName(selector), value!);

// 用法（属性重命名会编译报错，重构安全）
request.Conditions.AddFilter<SysUser, EnableStatus>(u => u.Status, status.Value);
request.Conditions.AddSort<SysUser>(u => u.CreatedTime, SortDirection.Descending, 1);
```
> 优先落地“方案二”的扩展方法集（`AddFilter<T,TValue>`/`AddSort<T>`/`SetKeyword<T>(params Expression...)`），全模块逐步替换裸 `nameof`，旧重载保留过渡。

### 3.2 查询范式边界规范（R-A / R-C）
写入团队规范并据此清理：

| 场景 | 用什么 | 理由 |
| --- | --- | --- |
| 列表/分页/筛选/排序（前端驱动） | `QueryConditions` + 类型安全扩展 | 动态、与前端参数直通、已成主流 |
| 单实体唯一键读 | 仓储 `GetByXxxAsync` | 简单直接 |
| 跨实体/状态机/有效性等**业务判定**（如“有效用户角色”“可用租户”） | `Specification` | 可复用、可单测、可在领域服务与查询服务共享 |
| 聚合整存整取 | 聚合仓储 `GetAggregate/SaveAggregate` | 维护聚合一致性 |

对现有 8 个 Specification：
- 若其判定逻辑在领域服务与查询/校验中**重复出现** → 推广为统一调用点；
- 若仅领域内一次性使用 → 保留即可，不强行铺开（避免“为规约而规约”）。

### 3.3 抽象增补（R-D，可选）
仅当确有跨仓储复用价值时增补，避免空泛：
```csharp
public interface ISaasRepository<TEntity> : IRepositoryBase<TEntity, long>
    where TEntity : class, IEntityBase<long>
{
    Task<List<TEntity>> GetByIdsAsync(IReadOnlyCollection<long> ids, CancellationToken ct = default);
}
```
在 `SaasRepository<T>` 基类实现一次，避免各仓储重复写 `ids.Contains` 查询。

### 3.4 聚合分类审计基线（R-E）
在本仓库维护“实体 → 仓储基类”对照表（节选）：

| 实体 | 是否聚合根 | 仓储基类 | 备注 |
| --- | --- | --- | --- |
| SysUser | 是 | AggregateRepository | 含安全/会话等关联 |
| SysRole | 是 | AggregateRepository | 含角色权限 |
| SysConstraintRule | 是 | AggregateRepository | 含 `Items` 子实体 |
| SysUserRole | 否 | Repository | 关系实体 |
| SysLoginLog | 否 | Repository | 只读日志 |
| …… | …… | …… | 新增实体须在此登记 |

规则：**新增实体前先判定聚合归属并登记**；CI 可加检查——实现 `IAggregateRoot` 的实体其仓储必须继承聚合基类。

## 4. 迁移步骤
1. 落地 `QueryConditions` 的类型安全扩展方法（`AddFilter<T,TValue>` 等）。
2. 写入“查询范式边界规范”，团队评审。
3. 试点 Identity/User：`UserQueryService` 裸 `nameof` 全部替换为类型安全调用。
4. 审计 8 个 Specification 的复用度，决定推广或保留。
5. （可选）`ISaasRepository` 增补 `GetByIdsAsync` 并下沉实现。
6. 建立聚合分类对照表 + CI 校验。
7. 按子域水平铺开字段类型安全化。

子域检查项：
- [ ] 该子域查询服务无裸 `nameof` 过滤/排序。
- [ ] 仓储命名仍为 `GetByXxx/ExistsXxx/GetPaged`。
- [ ] 复杂业务判定走 Specification 或集中扩展，不行内重复。

## 5. 测试策略
- **扩展方法单测**：`AddFilter<T>(u=>u.Status, ...)` 生成的字段名/条件正确。
- **Specification 单测**：内存集合 `Where(spec.ToExpression().Compile())` 验证命中集。
- **仓储集成测试**：每子域 ≥1 条分页查询断言 `PageResultDtoBase` 元数据。
- **CI 校验**：聚合实体 ⇄ 仓储基类一致性。
