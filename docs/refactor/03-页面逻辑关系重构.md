## 1. 现状（实证）

### 1.1 页面与 API 模块清单（实测）
- 视图（`src/views`）：
  - `system/`：user、role、permission、department、notification
  - `platform/`：tenant、edition、app、menu、dict、config
  - `log/`：login、operation、access、api、diff、permission、exception
  - `workbench/`：index；`_core/`：登录/仪表盘/错误页等
- API 模块（`src/api/modules`）：按 `{domain}/{resource}.ts` + `{resource}.types.ts` 细粒度拆分（如 `authorization/user-role.ts`、`identity/user-security.ts`、`audit/login-log.ts` 等），每资源一对文件。

### 1.2 前端类型按资源手工镜像后端 DTO（**核心问题**）
每个资源的 `*.types.ts` 手写对应后端 DTO 的 TS 接口（如 `UserDataScopeDetailDto`/`UserDataScopeGrantDto` 等），与后端 `Application/Dtos` 字段一一对应、人工维护。
后端 DTO 字段一变，需手工同步所有相关 `*.types.ts`——**漂移高发**（资源越细，需同步的文件越多）。

### 1.3 已有读/命令工厂，但被大量绕过（不一致）
`src/api/base.ts` **已提供** `createReadApi(controller, resource)`（`page`/`detail`）与 `createCommandApi(controller, resource)`（`create`/`update`），以及底层 `createDynamicApiClient`、`createPageRequestParams`、`appendDynamicApiParam`、`formatDynamicApiRouteValue`。
但现有工厂能力**不完整**（仅覆盖标准 `page/detail/create/update`，缺 `delete`、子聚合命令、自定义 action、按字段过滤），导致很多资源**直接绕过工厂、用裸 `createDynamicApiClient` 手写**：
```ts
// 实例：authorization/user-data-scope.ts —— 未用工厂，手搓
const userDataScopeQueryApi = createDynamicApiClient('UserDataScopeQuery')
const userDataScopeCommandApi = createDynamicApiClient('User')
export const userDataScopeApi = {
  grant: i => userDataScopeCommandApi.post('UserDataScope', i),
  list: (userId, onlyValid = false) => { const p = {}; appendDynamicApiParam(p, 'OnlyValid', onlyValid); return userDataScopeQueryApi.get(`UserDataScopes/${formatDynamicApiRouteValue(userId)}`, p) },
  // …… 每资源各写一遍
}
```
后果：工厂与裸客户端两套写法并存，分页参数/路由格式各处重复。

### 1.4 链路六段的实际分布
| 段 | 位置 | 状态 |
| --- | --- | --- |
| 实体 | `Domain/Entities` | ✅ 事实源 |
| 权限码 | `Domain/Permissions/SaasPermissionCodes`（强类型，1383 行，含 `.All`） | ✅ 已强类型，质量高 |
| 菜单 | `Infrastructure/Seeders/SaasMenuSeeder`（手工 21KB） | ⚠️ 手工，与视图/路由手工对齐 |
| 路由 | 前端 `packages/router` + `src/router`（静态 + 动态） | ⚠️ 组件路径映射手工 |
| 视图 | `src/views/**/index.vue` | ⚠️ 新增页面易漏配菜单/权限 |
| API | `src/api/modules/*.ts` | ⚠️ 手工镜像类型 + 样板 |

> 注意：与“纯漂移”不同，权限段已是强类型常量（优秀基础），可作为生成与校验的锚点。

## 2. 重点一：后端契约 → 前端类型代码生成

**消除 `identity.ts` 等手工类型镜像。**

方案：后端启用 OpenAPI/Swagger（动态 API 已具备），前端用 `openapi-typescript`（或等价工具）生成 `src/api/generated/types.ts`。

```
后端 DTO/Contracts ──(构建期)──► OpenAPI schema ──(openapi-typescript)──► 前端 types.ts
```
- 前端只 `import type { SaasUserListItemDto } from '@/api/generated'`，不再手抄。
- 生成纳入前端构建脚本（`scripts/`）与 CI；后端契约一改，前端类型自动更新、不一致即编译失败。
- 过渡：先对一个域（identity）切到生成类型，验证后铺开。

## 3. 重点二：补全并统一资源工厂，消除裸客户端样板

`base.ts` 已有 `createReadApi`/`createCommandApi`，但能力不全（缺 delete、子聚合命令、自定义 action、按字段过滤），故被大量绕过。动作是**扩展为完整的 `defineResource` 工厂并强制统一使用**（淘汰裸 `createDynamicApiClient` 写法），把“命令控制器 + 查询控制器 + 分页参数 + 过滤”声明化。

```ts
// api/factory.ts（新增）
export interface ResourceOptions<TQuery> {
  query: string                 // 查询控制器，如 'UserQuery'
  command: string               // 命令控制器，如 'User'
  pageAction?: string           // 默认 `${command}Page`
  detailAction?: string         // 默认 `${command}Detail`
  toParams?: (q: TQuery) => Record<string, unknown>  // 仅声明差异过滤字段
}

export function defineResource<TList, TDetail, TCreate, TUpdate, TQuery extends PageRequest>(o: ResourceOptions<TQuery>) {
  const q = createDynamicApiClient(o.query)
  const c = createDynamicApiClient(o.command)
  return {
    page: (query: TQuery) => {
      const params: DynamicApiParams = {}
      appendPageRequestParams(params, query)
      Object.entries(o.toParams?.(query) ?? {}).forEach(([k, v]) => appendDynamicApiParam(params, k, v))
      return q.get(o.pageAction ?? `${o.command}Page`, params).then(parsePageResult<TList>)
    },
    detail: (id: ApiId) => q.get<TDetail>(`${o.detailAction ?? `${o.command}Detail`}/${id}`),
    create: (i: TCreate) => c.post<TDetail>(o.command, i),
    update: (i: TUpdate) => c.put<TDetail>(o.command, i),
    remove: (id: ApiId) => c.delete(`${o.command}/${id}`),
  }
}
```
```ts
// api/modules/identity.ts（重构后，声明式，体量大幅下降）
export const userApi = defineResource<SaasUserListItemDto, SaasUserDetailDto, SaasUserCreateInput, SaasUserUpdateInput, UserPageQuery>({
  query: 'UserQuery', command: 'User',
  toParams: q => ({ Keyword: q.keyword, Status: q.status, Gender: q.gender }),  // 唯一手写差异
})
```
> 子聚合命令（授角色/重置密码等）按 02 分册拆分后，对应建独立资源或在 userApi 上挂方法。

## 4. 重点三：页面登记表 + 一致性校验（兜底零漂移）

把链路六段登记到**单一事实表**，菜单/权限种子由它派生，前端路由由它校验。

```csharp
// 后端：Application/Pages/PageDescriptor.cs（强类型登记）
public sealed record PageDescriptor(
    string Code,               // "system.user"
    string Title,              // i18n key
    string Entity,             // nameof(SysUser)
    string ViewPath,           // "system/user/index"
    string RoutePath,          // "/system/user"
    string PermissionPrefix,   // SaasPermissionCodes.User 前缀
    PageOperation[] Operations,// [List,Detail,Create,Update,Delete,...]
    string? MenuParent, int Order);
```

派生与校验关系：
```
PageDescriptor[]
  ├─→ SaasMenuSeeder         （生成菜单：Title/Route/View/Order/Parent）
  ├─→ 权限一致性校验          （PermissionPrefix×Operations ⊆ SaasPermissionCodes.All）
  ├─→ 前端 routes 校验        （每个 ViewPath 必须存在 index.vue）
  └─→ CI 校验脚本            （实体存在？控制器存在？菜单与登记表一致？）
```

实施策略（**先校验、后生成**，降风险）：
1. 第一步只做**校验脚本**：录入现状到 `PageDescriptor[]`，校验当前菜单/权限/视图/控制器互相对齐，修平存量漂移。
2. 第二步把 `SaasMenuSeeder` 改为**从登记表生成**（见 04 分册）。

## 5. 重点四：BFF 聚合页约定统一

明确两类页面，前端按约定取数：

| 页面类型 | 数据来源 | 约定 |
| --- | --- | --- |
| 标准 CRUD 页（dict/config/log…） | 单资源 `defineResource` | 直连资源 API + 05 的 CRUD 脚手架 |
| 复合管理页（用户/角色/权限中心） | `Contracts/System` 的 `IXxxManagementQueryService` | 一次聚合请求返回主列表 + 关联选项 + 统计，前端**不并发多原子 API** |
| 工作台 | `Contracts/Workbench/IWorkbenchQueryService` | 一次返回各卡片数据 |

`Contracts/System`、`Contracts/Workbench` 已存在，需补齐：所有复合页都走聚合服务，统一返回 `XxxPageViewDto`。

## 6. 路由与菜单联动
- 以**动态路由**（后端菜单驱动）为主；`resolveView` 的组件别名映射改由“登记表导出的 ViewPath 清单”生成，去掉手工维护。
- 静态路由仅留 `_core`（登录/错误/仪表盘骨架）。
- 前端构建期校验：登记表中每个 `ViewPath` 必须有对应 `index.vue`，否则构建失败。

## 附录 A · 目标菜单与页面登记（权威实例）

> 这是 §2 抽象 `PageDescriptor[]` 的**具体落地实例**：菜单种子（04 分册）从本表生成，前端路由/视图按本表校验。
> 关键设计原则：**一个实体未必一个页面**。强关联的子实体通过「同页标签页 / 同页双栏 / 同页状态切换」并入主页面，避免菜单与页面碎片化。

### A.2 由本表得出的设计要点
1. **页面合并模式三选一**，前端脚手架（05 分册）须支持：
   - **详情标签页**（主实体一行 → 抽屉/详情页内多 Tab 加载子实体）：用户、角色、应用、任务。
   - **同页多标签平铺**（无主从、并列资源）：权限管理、授权申请、邮件短信、日志查询。
   - **同页双栏/状态切换**：字典（左右栏）、审批中心（状态切换）。
2. **菜单 `PageDescriptor`** 需扩展承载「子标签清单」字段，例如：
   ```csharp
   public sealed record PageDescriptor(
       string Code, string Title, string Entity, string ViewPath, string RoutePath,
       string PermissionPrefix, PageOperation[] Operations,
       string? MenuParent, int Order,
       PageTab[]? Tabs = null);   // 子实体标签：每个 Tab 关联实体 + 权限前缀
   public sealed record PageTab(string Key, string Title, string Entity, string PermissionPrefix);
   ```
3. **权限前缀对齐**：每个页面与其子标签的 `PermissionPrefix` 必须能在 `SaasPermissionCodes` 找到对应分组（用户页的角色 Tab → `saas:user-role:*`），由 04 分册一致性校验覆盖。
4. **日志审计不重复**：`SysOpenApiLog`（接口调用日志）仅出现在「日志查询」标签内，不在开放平台重复建页。
5. **分组与排序**：分组顺序即菜单 `Order` 段位（工作台→身份权限→租户→消息→审批→文件→开放平台→系统设置→日志审计），菜单种子按此生成。
