# XiHan.BasicApp.Saas 重构方案 · 总览

> 版本：v2.0（已按真实代码校正） · 日期：2026-05-30
> 范围：`backend/src/modules/XiHan.BasicApp.Saas` 全模块 + `frontend` 对应页面
> 基准：以 `Domain/Entities`（约 145 个实体/枚举/扩展）为唯一领域事实来源。

> ⚠️ **重要前提**：经逐文件核对，本模块**当前已是相当成熟的 DDD + CQRS 架构**，并非“烂摊子重写”。本方案因此定位为**精炼式重构（Refinement）**而非推倒重来——目标是消除少数真实存在的不一致与重复、降低手工同步面、引入代码生成与统一脚手架。下文“现状”一栏均来自实际代码抽样，不含臆测。

本方案为分模块文档：

| 分册 | 主题 | 核心动作 |
| --- | --- | --- |
| [01-仓储层重构](./01-仓储层重构.md) | Repository | 统一查询范式（QueryConditions vs Specification）、字段引用类型安全、聚合分类审计 |
| [02-服务层重构](./02-服务层重构.md) | Service | 拆分超大“上帝服务”、校验入口归一、缓存失效覆盖审计、样板收敛 |
| [03-页面逻辑关系重构](./03-页面逻辑关系重构.md) | Page Logic | 前端类型/接口代码生成、资源工厂、实体→权限→菜单→路由→视图一致性校验 |
| [04-种子数据重构](./04-种子数据重构.md) | Seed Data | 权限/菜单生成式、字典与枚举对齐、系统/演示数据分层、幂等 |
| [05-UI美化重构](./05-UI美化重构.md) | UI | 抽取统一 CRUD 脚手架、设计令牌化、列表/表单 schema 驱动、暗色一致 |

## 1. 现状速写（事实基线）

### 1.1 技术栈
- 后端：.NET 10、DDD 分层、SqlSugar ORM、SignalR（`Hubs/`）、本地事件总线 `ILocalEventBus`、工作单元 `[UnitOfWork]`、动态 API `[DynamicApi]`、声明式鉴权 `[PermissionAuthorize]`。
- 框架底座 `XiHan.Framework`：
  - 仓储：`IRepositoryBase<TEntity,TKey>` / `IAggregateRootRepository<TAgg,TKey>`，实现 `SqlSugarRepositoryBase` 系列；
  - 实体：`EntityBase<TKey>`（主键 `BasicId`、并发 `RowVersion`）、审计/多租户/软删基类；
  - 分页：`PageResultDtoBase<T>`（含 `.Map()`）+ 动态查询 `QueryConditions`（`SetKeyword`/`AddFilter`/`AddSort`）；
  - DI：`ITransientDependency`/`IScopedDependency`/`ISingletonDependency` 自动注册；
  - 种子：`IDataSeeder`（`Order`/`Name`/`SeedAsync`）。
- 前端：Vue 3.5 + Naive UI 2.44 + Pinia + Vue Router 4 + Vite 6 + Tailwind 3 + TS + vue-i18n；自有 monorepo（`packages/*`）；CSS 变量（HSL）+ `darkMode:'class'` 设计令牌；动态/静态双路由。

### 1.2 后端分层（实测，已较规范）
```
Domain/
  Entities/        SysXxx.cs(+partial) + Aggregates/ + Enums/ + Expands/
  Repositories/    Abstractions/(ISaasRepository, ISaasAggregateRepository) + 10 子域接口
  DomainServices/  各子域 IXxxDomainService + Implementations/（承载业务规则、不变量、领域事件）
  Specifications/  8 个规约（仅在 DomainServices 内使用）
  Permissions/     SaasPermissionCodes.cs（强类型权限码，1383 行，含 .All 集合）
  Events/ ValueObjects/ Configurations/
Application/
  AppServices/     命令服务（→ DomainService）
  QueryServices/   查询服务（→ Repository.GetPagedAsync(QueryConditions)）
  Contracts/       IXxxAppService（命令）/ IXxxQueryService（查询）+ System/ + Workbench/(BFF)
  Dtos/  Mappers/(50 个显式静态映射器，0 处 Adapt)  Caching/  EventHandlers/(事件驱动缓存失效)
Infrastructure/
  Repositories/    50 实现（SaasRepository<T> / SaasAggregateRepository<T>）
  Seeders/         11 个（SaasPermissionSeeder 达 63KB）
  Auth/ Logging/ Messaging/ Security/ Tasks/
Hubs/
```

### 1.3 已经做得好的地方（重构需保留）
- ✅ **CQRS 清晰**：命令/查询物理分离；命令侧 `AppService → DomainService → Repository`，查询侧 `QueryService → Repository`。
- ✅ **映射零魔法**：100% 显式静态映射器（`UserApplicationMapper.ToDetailDto` 等），全模块 **0 处 `.Adapt<>`**，可读、可调试、可控。
- ✅ **缓存事件驱动**：业务服务不手工清缓存；由 `EventHandlers/*` 监听领域事件后调 `ISaasCacheInvalidator`。
- ✅ **权限强类型**：`SaasPermissionCodes`（`saas:user:create` 等）+ `[PermissionAuthorize(...)]`，并提供 `.All` 全集。
- ✅ **仓储较薄**：如 `IUserRepository` 仅 `GetByUserNameAsync` / `ExistsUserNameAsync`，分页统一 `GetPagedAsync` + `PageResultDtoBase`。
- ✅ **聚合区分**：聚合根用 `ISaasAggregateRepository<T>`（User/Role/Permission/Operation/Resource/OAuthApp/Department/Tenant/Review/Task/ConstraintRule），普通实体用 `ISaasRepository<T>`。

## 2. 真实问题诊断（核对后保留项）

> 每条均有实证；夸大的“伪问题”已剔除。详见各分册。

### P1 查询范式二元并存（仓储/服务）
- 列表查询主流走**动态 `QueryConditions`**（30 个 QueryService 使用 `SetKeyword/AddFilter/AddSort`），字段名以 `nameof(SysUser.UserName)` 字符串传入；
- 同时存在 **8 个强类型 `Specification`**，但 **应用层 0 处引用**，仅 `DomainServices` 内部使用。
- 后果：两套查询心智模型并存；动态范式丢失编译期字段安全（重命名实体属性不报错）。

### P2 “上帝服务”与超大接口（服务）
- `PermissionAppService` 532 行、`UserAppService` 439 行、`ProfileAppService` 423 行。以 `UserAppService` 为例，单个服务/接口聚合了：用户核心 + 安全 + 角色 + 直授权限 + 数据范围 + 部门 + 会话 共 7 类职责、20+ 方法。
- 后果：单类职责过宽、接口臃肿、变更影响面大、并行开发易冲突。

### P3 前端类型/接口手工镜像后端（页面逻辑）
- 前端 API 按 `{domain}/{resource}.ts` + `{resource}.types.ts` 细粒度拆分，每个 `*.types.ts` **手工镜像**后端对应 DTO 字段；后端 DTO 一变需手工同步多个文件。
- `api/base.ts` 仅提供底层 `createDynamicApiClient`/`appendPageRequestParams`，**无资源工厂**，每个资源各写一遍 client + 参数追加。
- 后果：后端 DTO 一改，前端类型需手工跟改；样板量巨大、易漂移。

### P4 种子手工巨型且半硬编码（种子数据）
- `SaasPermissionSeeder.cs` 63KB：引用强类型 `SaasPermissionCodes` 354 次，**同时**散落 177 处硬编码 `"saas..."` 字符串——两套并存、易脱节；
- `SaasMenuSeeder.cs` 21KB 纯手工，与前端视图/路由手工对齐；
- 字典种子与 `Enums/SysXxx.Enum.cs` 语义重复，两处各维护。
- 已有 `SaasPermissionCodes.All` 可作为生成源，但当前未被用于自动产出权限种子。

### P5 列表页为超大单文件、无共享脚手架（UI）
- `views/system/user/index.vue` **1968 行**（message 776 行等同样庞大）：在单文件内手搓 表格 + 分页 + 弹窗 + 抽屉 + 多 Tab 表单 + 列定义 + 状态。各页面各写一套搜索区/空态/加载态/暗色处理。
- 后果：风格不统一、维护成本高、复制粘贴扩散。

### P6 缓存失效覆盖待审计（服务，次要）
- 12 个 EventHandler 中仅 4 个调用 `CacheInvalidator`（Authorization/DataScope/FieldLevelSecurity/Hierarchy）。需确认其余写操作是否都有对应失效路径（可能正确、也可能存在遗漏）。

## 3. 目标架构（北极星，增量演进）

```
实体 Domain/Entities  ← 唯一事实来源
   │
   ├─ 仓储层：薄仓储 + 统一查询范式（见 01：QueryConditions 为主、Specification 复用于复杂领域判定，字段引用类型安全化）
   ├─ 领域服务：业务规则/不变量/领域事件（保留并强化）
   ├─ 应用层：命令服务（按子聚合拆分） / 查询服务；映射(显式) / 校验(归一) / 缓存(事件驱动，补全覆盖)
   ├─ 契约 → 控制器 → 前端 api（由 OpenAPI 生成 TS 类型 + 资源工厂消除样板）
   ├─ 页面链路：实体→权限码→菜单→路由→视图 由“页面登记表 + CI 校验”兜底零漂移
   ├─ 种子：权限/菜单从 SaasPermissionCodes/登记表生成；系统数据 vs 演示数据分层；全幂等
   └─ UI：统一 CRUD 脚手架（schema 驱动表格/表单）+ 设计令牌（亮暗一致）
```

> ⚠️ **全局约定（重建模式）**：线上数据库将完全重建。本方案**不实现任何旧数据兼容、数据迁移或存量清理逻辑**；所有种子/改动按全新空库假设编写，只需保证幂等与正确性。

设计原则：
1. **保留既有优点**（显式映射、事件驱动缓存、CQRS、强类型权限），不为“重构”而破坏。
2. **降低手工同步面**：能生成的（前端类型、权限/菜单种子）就生成。
3. **统一二元范式**：查询、UI 各收敛到一套主路径。
4. **拆宽为窄**：超大服务/接口/视图按职责切分。
5. **可校验**：链路一致性、权限-枚举一致性进 CI。

## 4. 命名与约定基线

| 维度 | 约定（多数已落地，少数为新增统一） |
| --- | --- |
| 实体主键 | `BasicId`（实体与 DTO 均沿用，保持现状一致） |
| 命令服务 | `XxxAppService : SaasApplicationService`，实现 `IXxxAppService`；**按子聚合拆分** |
| 查询服务 | `XxxQueryService : SaasApplicationService`，实现 `IXxxQueryService` |
| 仓储读 | `GetByXxxAsync`（唯一键，可空）/`ExistsXxxAsync`/`GetPagedAsync`/`GetByIdAsync`（保持现状命名） |
| 映射 | 子域级显式静态 `XxxApplicationMapper`（保持现状，禁止引入 `Adapt`） |
| 缓存 | 仅经 `EventHandlers` → `ISaasCacheInvalidator`（保持现状，补全覆盖） |
| 查询字段引用 | 用强类型 selector/常量替代裸 `nameof` 字符串（新增，见 01） |
| 权限码 | `SaasPermissionCodes`（保持），种子从 `.All` + 元数据生成（新增） |
| 前端类型 | 由后端 OpenAPI 生成，禁止手工镜像（新增，见 03） |
| 前端资源 API | 资源工厂 `defineResource`（新增，见 03） |
| 前端视图 | `views/{group}/{feature}/index.vue` + 共享 CRUD 脚手架（新增，见 05） |

## 6. 风险与对策
| 风险 | 等级 | 对策 |
| --- | --- | --- |
| 拆服务影响动态 API 路由/前端调用 | 中 | 控制器分组名保持稳定；拆分时保留旧 action 路径一个迭代 |
| 查询范式切换回归 | 中 | 范式以“新写用类型安全、旧代码渐迁”推进，不一刀切 |
| 种子生成与历史数据冲突 | 中 | 生成式种子一律按 code/key Upsert；系统/演示数据分文件 |
| 前后端契约漂移 | 中 | OpenAPI 生成 + CI 一致性校验 |
| UI 脚手架抽象过度 | 低 | 先沉淀 3 个真实页面共性再抽象，避免过早抽象 |
