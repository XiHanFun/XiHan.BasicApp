## 1. 现状（实证）

### 1.1 种子体系（已有良好骨架）
- 统一实现 `ISaasDataSeeder`，排序由集中常量 `SaasDataSeederOrder.*` 控制（如 `SaasDataSeederOrder.Permission/Menu/Dict`）。✅ 排序与执行契约清晰。
- 11 个种子：Configuration、DemoUser、Department、Dict、IdentityPermission、Identity、Menu、Notification、Permission(63KB)、StorageConfig、TenantEdition。

### 1.2 权限种子：强类型码 + 手工元数据（**核心问题**）
```csharp
// SaasPermissionSeeder.cs
private static SysPermission CreatePermission(string code, string name, PermissionType type, ...);
// 逐条手写：
CreatePermission(SaasPermissionCodes.Tenant.Read, "查看租户", PermissionType.Operation, ...),
CreatePermission(SaasPermissionCodes.Tenant.Create, "新增租户", PermissionType.Operation, ...),
// …… 数百条
```
- 权限**码**已强类型（引用 `SaasPermissionCodes` 354 次），但**显示名/类型/父级/排序等元数据**逐条手写；另有 177 处硬编码 `"saas..."` 字符串（多为资源/父级引用）。
- 风险：`SaasPermissionCodes.All` 与种子表两处维护，**新增权限码可能漏种子**（或反之）。

### 1.3 字典种子与枚举重复
`SaasDictSeeder` 手工 `BuildDicts()`，其字典项与 `Domain/Entities/Enums/SysXxx.Enum.cs` 语义重复（如状态、性别、渠道），两处各维护、易不一致。

### 1.4 系统数据与演示数据
`SaasDemoUserSeeder`(19KB) 为演示数据，与系统必需数据（权限/菜单/字典）混在同一种子目录，缺少“环境开关”与清晰分层。

## 2. 目标设计

### 2.1 权限种子生成化：单一注解源（最高收益）
把权限的**元数据**附着到权限码定义旁，使 `SaasPermissionCodes` 成为唯一源，种子全自动产出。

```csharp
// Domain/Permissions/SaasPermissionDefinition.cs（新增：码 + 元数据一处定义）
public sealed record SaasPermissionDefinition(
    string Code, string DisplayName, PermissionType Type,
    string? ParentCode = null, int Order = 0, string? ResourceCode = null);

public static class SaasPermissionDefinitions
{
    public static readonly IReadOnlyList<SaasPermissionDefinition> All =
    [
        new(SaasPermissionCodes.Tenant.Read,   "查看租户", PermissionType.Operation, ResourceCode: "saas:tenant"),
        new(SaasPermissionCodes.Tenant.Create, "新增租户", PermissionType.Operation, ResourceCode: "saas:tenant"),
        // …… 由现有 CreatePermission 调用机械迁移而来
    ];
}
```
```csharp
// SaasPermissionSeeder 重构后：纯遍历 + Upsert，无逐条字面量
public async Task SeedAsync()
{
    var entities = SaasPermissionDefinitions.All.Select(ToEntity).ToList();
    await _permissionRepository.UpsertRangeAsync(entities, x => x.Code);   // 幂等
}
```
一致性校验（进 CI）：
```
assert SaasPermissionCodes.All 的集合 == SaasPermissionDefinitions.All.Select(d=>d.Code) 的集合
assert 种子落库后的权限码集合 == 上述集合
```
> 这样“新增一个权限码却忘了配元数据/种子”会在编译或 CI 阶段暴露。

### 2.2 菜单种子以页面登记表为源（衔接 03）
`SaasMenuSeeder` 改为遍历 `PageDescriptor[]` 生成菜单（Title/Route/ViewPath/Parent/Order/关联权限）：
```csharp
public async Task SeedAsync()
{
    var menus = PageRegistry.All.Select(ToMenuEntity).ToList();
    await _menuRepository.UpsertRangeAsync(menus, x => x.Code);
}
```
消除“菜单 21KB 手工 ⇄ 前端视图 ⇄ 路由”的三方手工对齐。

### 2.3 字典与枚举对齐
两种策略择一并统一：
- **策略 A（推荐）**：枚举为源，字典种子从枚举反射生成（枚举值 + `[Description]` → 字典项）。适合“状态/性别/渠道”等与代码强绑定的枚举。
```csharp
foreach (var v in Enum.GetValues<EnableStatus>())
    items.Add(new SysDictItem { DictCode = "enable_status", Value = ((int)v).ToString(), Label = v.GetDescription() });
```
- **策略 B**：纯业务字典（与代码无关、运营可改）保持手工种子。
> 划清：与枚举绑定 → 生成；纯运营配置 → 手工。避免“枚举改了字典没改”。

### 2.4 系统数据 vs 演示数据分层
- 目录/命名分层：`Seeders/System/*`（权限/菜单/字典/版本，生产必需）与 `Seeders/Demo/*`（演示用户/示例租户）。
- 环境开关：演示种子仅在 `Development`/显式开关下执行；生产默认跳过。
- `SaasDataSeederOrder` 保持集中；演示种子排在系统种子之后。

## 3. 幂等规范（全种子统一）
- 一律按**业务键 Upsert**（权限按 `Code`、菜单按 `Code`、字典项按 `DictCode+Value`、用户按 `UserName`），禁止“先清空再插入”。
- 重复执行不产生重复行、不破坏已被引用的数据（外键安全）。
- 删除处理：若某权限码被移除，提供**显式停用**而非物理删除（避免误删被角色引用的权限），或在校验脚本中报告“孤立权限”。
