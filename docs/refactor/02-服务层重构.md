## 1. 现状（实证）

### 1.1 命令服务：薄编排，委托领域服务（`UserAppService`，已是好模式）
```csharp
[Authorize, DynamicApi(Group="BasicApp.Saas", Tag="用户")]
public sealed class UserAppService(IUserDomainService userDomain, ICurrentUser currentUser, ILocalEventBus bus)
    : SaasApplicationService, IUserAppService
{
    [UnitOfWork(true)]
    [PermissionAuthorize(SaasPermissionCodes.User.Create)]
    public async Task<UserDetailDto> CreateUserAsync(UserCreateDto input, CancellationToken ct = default)
    {
        ArgumentNullException.ThrowIfNull(input);
        ct.ThrowIfCancellationRequested();
        var result = await userDomain.CreateUserAsync(UserApplicationMapper.ToCreateCommand(input, currentUser.UserId), ct);
        return UserApplicationMapper.ToDetailDto(result.User);
    }
    // ……
}
```
✅ 业务规则在领域服务、映射显式、鉴权/事务声明式、领域事件经 `ILocalEventBus`。**这套模式予以保留。**

### 1.2 查询服务：QueryConditions 构建（`UserQueryService`）
`AppService` 不做查询；`QueryService` 用 `QueryConditions` 构请求 → `GetPagedAsync` → `PageResultDtoBase.Map(Mapper)`。结构清晰。

## 2. 重点一：拆分上帝服务（S-A / S-B）

### 2.1 按“子聚合 / 关注点”垂直切分
以 User 为例，将 1 个 439 行服务 + 1 个 20+ 方法接口，拆为**按子聚合**的多服务多接口：

| 拆分后服务 | 接口 | 职责 | 权限前缀 |
| --- | --- | --- | --- |
| `UserAppService` | `IUserAppService` | 用户核心：增删改、状态 | `saas:user:*` |
| `UserSecurityAppService` | `IUserSecurityAppService` | 重置密码、锁定、登录策略 | `saas:user-security:*` |
| `UserRoleAppService` | `IUserRoleAppService` | 授/撤角色、更新、状态 | `saas:user-role:*` |
| `UserPermissionAppService` | `IUserPermissionAppService` | 直授权限增删改状态 | `saas:user-permission:*` |
| `UserDataScopeAppService` | `IUserDataScopeAppService` | 数据范围授予 | `saas:user-datascope:*` |
| `UserDepartmentAppService` | `IUserDepartmentAppService` | 部门归属 | `saas:user-department:*` |
| `UserSessionAppService` | `IUserSessionAppService` | 会话撤销 | `saas:user-session:*` |

要点：
- 权限码已天然按子聚合分组（`SaasPermissionCodes.UserRole.Grant` 等），拆分**顺理成章**。
- 领域服务 `IUserDomainService` 可保持不变（或同步拆为 `IUserRoleDomainService` 等，视其体量）。
- 动态 API `[DynamicApi(... Tag="用户角色")]` 用不同 Tag，控制器分组更清晰；**保留原 action 路径一个迭代**避免前端断链。

对 `PermissionAppService`(532) / `ProfileAppService`(423) 同理按子关注点拆分。

### 2.2 拆分判据（避免过度拆分）
- 一个服务对应**一个子聚合或一个清晰用例族**；
- 行数经验阈值：命令服务 > ~250 行或方法 > ~8 个 → 评估拆分；
- 拆分以“是否能独立鉴权、独立演进”为准，而非单纯按行数。

## 3. 重点二：校验入口归一（S-C）

明确三层校验职责：

| 层级 | 放什么 | 载体 |
| --- | --- | --- |
| 形态校验 | 必填、长度、范围、格式 | DTO 上 DataAnnotations 或 FluentValidation，统一在管道层执行 |
| 应用级前置 | 入参非空、`ct` 检查 | 收敛到命令基类/小工具方法，减少重复样板 |
| 业务校验 | 唯一性、状态可达、配额 | 领域服务（首选）或 `IXxxValidator`，**单一出处** |

动作：
- 审计 `SaasCommandValidation` 当前承担的职责，归类到上表；纯形态校验迁到 DTO 注解，业务校验下沉领域服务。
- 统一异常：`UserFriendlyException` + 错误码常量表（便于前端 i18n），替代散落的字符串消息。

## 4. 重点三：缓存失效覆盖审计（S-D）

现状缓存为**事件驱动**（优点，保留）：写操作发领域事件 → `EventHandlers/*` → `ISaasCacheInvalidator`。

动作：
1. 列出所有**会影响缓存快照**的写操作（用户/角色/权限/菜单/数据范围/层级/配置 等）。
2. 对照 12 个 EventHandler，确认每个写操作都有对应失效路径；补齐遗漏的事件与 handler。
3. 形成“写操作 → 领域事件 → 失效缓存键”矩阵文档，纳入评审与 CI 检查（新增写操作须登记）。

```
示例矩阵
写操作                    领域事件                         失效缓存
授予用户角色              AuthorizationChangedEvent        SaasAuthorizationSnapshot(userId)
更新菜单                  MenuChangedEvent                 SaasMenuRoutes(*)         ← 若缺则补
更新配置                  ConfigChangedEvent               SaasConfigValue(key)      ← 若缺则补
```

## 5. 重点四：样板收敛（S-E，温和）

在不破坏“显式鉴权/事务特性”的前提下，收敛重复前置代码：

```csharp
// SaasApplicationService 增加受保护小工具
protected static void EnsureInput<T>(T input, CancellationToken ct) where T : class
{
    ArgumentNullException.ThrowIfNull(input);
    ct.ThrowIfCancellationRequested();
}
```
```csharp
public async Task<UserDetailDto> CreateUserAsync(UserCreateDto input, CancellationToken ct = default)
{
    EnsureInput(input, ct);
    var r = await userDomain.CreateUserAsync(UserApplicationMapper.ToCreateCommand(input, currentUser.UserId), ct);
    return UserApplicationMapper.ToDetailDto(r.User);
}
```
> 注意：`[UnitOfWork]`/`[PermissionAuthorize]`/`[DynamicApi]` 等特性**保持每方法显式**，不要为减行数而隐藏鉴权语义。映射保持显式静态调用（**不要引入 Adapt**）。

## 6. 不做什么（避免过度工程）
- ❌ 不引入通用 `CrudAppService` 强行套用——本模块多数命令是“领域命令”（授予/撤销/状态流转）而非纯 CRUD，强套基类反而别扭。
- ❌ 不用反射/源生成替换显式映射器——显式映射是本项目刻意的可读性选择。
- ❌ 不把事件驱动缓存改回服务内手工清理。
