# 02 · 附录 — 上帝服务拆分对照清单（任务 I 实施依据）

> 上游：[02-服务层重构](./02-服务层重构.md)
> 本清单基于对真实代码的逐文件核对（2026-05-30），用于在动手前对齐拆分边界、前端路由影响与领域服务复用关系。
> ⚠️ **重建约定**：线上库与前端将完全重建，不保留旧 action 路由别名；拆分后前端 API 同步改指向新控制器即可。

---

## 0. 关键事实（核对结论）

| 事实 | 证据 |
| --- | --- |
| 三个上帝服务无后端内部调用者 | `grep IUserAppService/IPermissionAppService/IProfileAppService` 仅出现在契约与控制器自动装配，无业务代码直接依赖 → **可安全拆分/改签名** |
| 方法名与领域服务方法同名但分属两层 | 如 `CreateUserRoleAsync` 在 AppService（编排）与 `IUserDomainService`（业务）各有一份，拆 AppService 不影响领域层 |
| 前端 API 已按子聚合分文件，但命令端全部指向单一控制器 | `user-role.ts`→`createDynamicApiClient('User')`、`resource.ts`→`'Permission'`，拆后需改为各自新控制器名 |
| 领域服务的拆分粒度不一 | Identity 侧为单体 `IUserDomainService`；Authorization 侧已细分为 10 个领域服务（`IPermissionCatalogDomainService` 等）。**拆 AppService 时领域服务保持不动** |
| 动态 API 路由 = 控制器名(去 AppService 后缀) + action 名(去 Async) | 见 `DefaultDynamicApiConvention`：`UserRoleAppService.CreateUserRoleAsync` → 控制器 `UserRole` + action `CreateUserRole` |

## 1. UserAppService（439 行 / 24 方法 / 7 region）→ 拆 7 服务

依赖：`IUserDomainService`（全部方法委托它）、`ICurrentUser`（仅核心创建/会话）、`ILocalEventBus`（仅会话发事件）。

| 拆分后服务 / 接口 | 来源 region | 方法 | 权限码前缀 | 依赖 | 前端命令文件现指向 → 应改为 |
| --- | --- | --- | --- | --- | --- |
| `UserAppService` / `IUserAppService` | 用户核心 | Create/Update/UpdateStatus/Delete (4) | `User.*` | UserDomain + CurrentUser | `identity/user.ts` `'User'` →（不变）|
| `UserSecurityAppService` | 用户安全 | ResetPassword/UpdateLock/UpdateLoginPolicy (3) | `UserSecurity.*` | UserDomain | `identity/user-security.ts` `'User'` → `'UserSecurity'` |
| `UserRoleAppService` | 用户角色 | Create/Update/UpdateStatus/Delete (4) | `UserRole.*` | UserDomain | `authorization/user-role.ts` `'User'` → `'UserRole'` |
| `UserPermissionAppService` | 用户直授权限 | Create/Update/UpdateStatus/Delete (4) | `UserPermission.*` | UserDomain | `authorization/user-permission.ts` `'User'` → `'UserPermission'` |
| `UserDataScopeAppService` | 用户数据范围 | Create/Update/UpdateStatus/Delete (4) | `UserDataScope.*` | UserDomain | `authorization/user-data-scope.ts` `'User'` → `'UserDataScope'` |
| `UserDepartmentAppService` | 用户部门 | Create/Update/UpdateStatus/Delete (4) | `UserDepartment.*` | UserDomain | `organization/user-department.ts` `'User'` → `'UserDepartment'` |
| `UserSessionAppService` | 用户会话 | RevokeSession/RevokeSessions (2) | `UserSession.*` | UserDomain + CurrentUser + EventBus | `identity/user-session.ts` `'User'` → `'UserSession'` |

> 注意：核心 region 的 `ResetUserPasswordAsync` 实际返回 `UserSecurityDetailDto`，归入 `UserSecurityAppService`（已在上表归类）；前端 `user.ts` 的 `resetPassword` 调用需相应迁到 user-security。

## 2. PermissionAppService（532 行 / 24 方法 / 6 region）→ 拆 6 服务

每个子聚合 4 方法（Create/Delete/Update/UpdateStatus），分别委托对应领域服务。

| 拆分后服务 / 接口 | 来源 region | 委托领域服务 | 权限码前缀 | 前端命令文件现指向 → 应改为 |
| --- | --- | --- | --- | --- |
| `PermissionAppService` / `IPermissionAppService` | Permission | `IPermissionCatalogDomainService` | `Permission.*` | `permission.ts` `'Permission'` →（不变）|
| `ResourceAppService` | Resource | `IPermissionCatalogDomainService` | `Resource.*` | `resource.ts` `'Permission'` → `'Resource'` |
| `OperationAppService` | Operation | `IPermissionCatalogDomainService` | `Operation.*` | `operation.ts` `'Permission'` → `'Operation'` |
| `PermissionConditionAppService` | PermissionCondition | `IPermissionGovernanceDomainService`* | `PermissionCondition.*` | `permission-condition.ts` `'Permission'` → `'PermissionCondition'` |
| `PermissionDelegationAppService` | PermissionDelegation | `IPermissionDelegationDomainService` | `PermissionDelegation.*` | `permission-delegation.ts` `'Permission'` → `'PermissionDelegation'` |
| `PermissionRequestAppService` | PermissionRequest | `IPermissionRequestDomainService` | `PermissionRequest.*` | `permission-request.ts` `'Permission'` → `'PermissionRequest'` |

> *委托关系以 `PermissionAppService` 当前构造函数注入为准，实施时逐 region 照搬其原有委托调用，不改领域层。

## 3. ProfileAppService（423 行 / 24 方法）→ 评估：保留为门面，按关注点分部分类

Profile 是**当前登录用户自助 BFF**，24 方法均围绕"我"操作，无子聚合实体边界，依赖 `ICurrentUser` + `IProfileDomainService` + `IUserDomainService` + `IUserProfileQueryService`。

按 02 正文"拆分判据（独立鉴权/独立演进）"，Profile **不宜按实体硬拆**，建议二选一（实施时定）：

- **方案 A（推荐，低风险）**：保留单一 `ProfileAppService`，但用 `partial class` 按关注点分文件（`ProfileAppService.Account.cs` / `.Security.cs` / `.Verification.cs` / `.Session.cs`），降低单文件体量、不改路由、不改前端。
- **方案 B**：拆 `ProfileSecurityAppService`（2FA/密码）、`ProfileVerificationAppService`（邮箱/手机验证码）、`ProfileSessionAppService`（会话）、`ProfileAppService`（资料/账号），前端 `profile/profile.ts`（现指向 `'Profile'`）随之拆分。

关注点分组（供两方案共用）：
| 分组 | 方法 |
| --- | --- |
| 账号资料 | GetProfile/UpdateProfile/ChangeUserName/DeactivateAccount/DeleteAccount/GetLinkedAccounts/UnlinkAccount |
| 安全(密码/2FA) | ChangePassword/Enable2FA/Disable2FA/Setup2FA/Send2FASetupCode |
| 验证(邮箱/手机) | SendChangeEmailCode/ConfirmChangeEmail/SendEmailVerifyCode/VerifyEmail/SendChangePhoneCode/ConfirmChangePhone/SendPhoneVerifyCode/VerifyPhone |
| 会话/登录日志 | GetSessions/RevokeSession/RevokeOtherSessions/GetLoginLogs |

## 4. 实施顺序（每步独立编译 + 提交）

1. **User 垂直切片**（样板）：建 6 个新服务+接口（Security/Role/Permission/DataScope/Department/Session），各从原 `UserAppService` 搬运对应 region；`UserAppService` 仅留核心 4 方法；`IUserAppService` 同步瘦身。`dotnet build` 通过即提交。
2. 前端同步：6 个 user-* 命令文件改 `createDynamicApiClient` 指向新控制器；`pnpm build`/类型检查通过即提交。
3. **Permission 拆 6**：同上模式，逐子聚合搬运 + 前端改指向。
4. **Profile**：按方案 A 或 B 落地。
5. 每个服务保留 `[UnitOfWork]`/`[PermissionAuthorize]`/`[DynamicApi(Tag=...)]` 显式特性；映射保持显式静态；不引入 Adapt。

## 5. 验收
- [ ] 无 > ~250 行命令服务；每服务对应单一子聚合/关注点。
- [ ] 拆分后 `dotnet build` 0 错误；前端类型检查通过。
- [ ] 前端命令客户端指向各自新控制器，调用链端到端通。
- [ ] 领域层、显式映射、事件驱动缓存零改动。
