> 以 `Domain/Entities` 为唯一领域事实来源，对仓储、服务、页面逻辑、种子数据、UI 进行**精炼式重构**。

## 一句话结论
本模块当前已是成熟的 DDD + CQRS 架构（显式映射、事件驱动缓存、强类型权限、聚合区分均已就位）。本方案不推倒重来，而是消除少数真实的不一致与重复（查询范式二元、上帝服务、前端类型手抄、半硬编码种子、巨型单文件页面），并引入代码生成与统一脚手架，降低长期维护成本。

## 建议落地次序
S0 决策对齐 → S1 服务拆分 → S2 查询统一 → S3 种子生成化 → S4 前端类型生成 + 资源工厂 → S5 UI 脚手架。
建议以 **Identity/User 垂直切片**贯穿验证后再水平铺开。
