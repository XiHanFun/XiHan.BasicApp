## 1. 背景与问题诊断

### 1.1 表象
平台超级管理员在菜单管理页编辑任意菜单 → 后端抛 `InvalidOperationException：平台级全局菜单必须通过平台运维流程维护。`

### 1.2 根因链
```
超管登录默认进入"租户1"上下文（ICurrentTenant.Id == 1，非平台态）
   └─ 菜单 GetByIdAsync 走带租户过滤的查询，过滤条件 TenantId == 1 OR == 0，能查到全局菜单(TenantId=0)
        └─ 进入 MenuDomainService.GetEditableMenuOrThrowAsync
             └─ menu.IsGlobal (= TenantId==0) 命中
                  └─ 无条件抛错（不区分平台态/租户态、不区分角色）
```

### 1.3 结论
- **实体设计无误**：`TenantId=0` 作平台全局模板、`IsGlobal => TenantId==0` 派生、查询合并 `IN(0,current)`，是贯穿全框架的 Model A 设计，自洽且规范。
- **拦截是刻意的架构不变量**：相同的"平台级全局 XXX 必须通过平台运维流程维护"出现在约 10 个领域服务（见 §2.3），菜单是**遵守**该约定，并非异类。
- **真正的缺口有四个**（本方案要补的）：
  1. 领域闸门是**无条件**的——即使平台超管也被挡（缺"平台态放行"）。
  2. **没有平台态入口**——超管无法切换到无租户上下文去走"平台运维流程"。
  3. **租户开通流程未接线**——`TenantProvisionDomainService` 写好却无人调用，新建租户=空壳。
  4. **套餐(Edition)运行时未生效**——版本权限白名单只存表，鉴权/菜单可见性不按版本过滤。

## 3. 目标架构与设计主线

```
┌─ 平台态 (ICurrentTenant.Id == null)  ── 平台超管/平台管理员 ──┐
│   维护全局模板：菜单 / 权限 / 角色 / 操作 / 资源 / 版本 / 版本权限   │
│   SysTenantEditionPermission = 各版本对外开放的权限白名单           │
└───────────────────────────────────────────────────────────────┘
                         │ 租户订阅一个版本 SysTenant.EditionId
                         ▼
┌─ 租户态 (ICurrentTenant.Id == N>0) ───────────────────────────┐
│  开通时：建管理员(SysTenantUser) + 建租户角色 + 按版本白名单授权     │
│  运行时：有效权限 = 用户角色/直授权限 ∩ 版本白名单 （超管 * 例外）   │
│  菜单可见性：菜单.PermissionId 在有效权限内才可见（共享读 TenantId=0） │
│  租户可叠加自己的私有菜单/角色 (TenantId=N)                         │
└───────────────────────────────────────────────────────────────┘
                         │ SysTenantUser 多租户成员
                         ▼
        用户：登录前选租户 / 登录后切租户 / （平台超管）切到平台态
```

**四条原则**
1. 不改 Model A 底座；全局模板"共享读"，不做按租户克隆。
2. 平台运维不变量保留；只把"无条件拦截"升级为"平台态放行"。
3. 套餐门控以**复用权限**为主（菜单可见性由 PermissionId 推导），避免新增冗余关联表。
4. 所有改动可按 REQ 项追溯、回滚、验证。

## 6. Phase 1 平台运维闸门

**目标**：把无条件拦截升级为"平台态放行"，统一所有相关领域服务；解最初报错（配 Phase 2 入口）。

### 后端
- **REQ-1.1** 新增统一平台态判定。
  - 方式 A（推荐）：领域服务注入 `ICurrentTenant`，判 `IsPlatformOps = currentTenant.Id is null`。
  - 方式 B：抽象 `IPlatformOperationContext { bool IsPlatformOps; }`，实现内聚 `ICurrentTenant`(+ 可选 `ICurrentUser` 校 super_admin/PlatformAdmin)，各领域服务依赖它。
  - 文件：新增 `Domain/DomainServices/Shared/IPlatformOperationContext.cs` + 实现；或直接在各服务构造注入 `ICurrentTenant`。
- **REQ-1.2** 改造 §2.3 全部守卫：`if (x.IsGlobal) throw` → `if (x.IsGlobal && !IsPlatformOps) throw`；系统角色类同理（`role.RoleType == System && !IsPlatformOps`）。
  - 文件（逐一）：`MenuDomainService.cs:348`、`PermissionCatalogDomainService.cs:726/744/762`、`RoleDomainService.cs:605/683/909`、`ConstraintRuleDomainService.cs:160/407`、`FieldLevelSecurityDomainService.cs:342`、`PermissionDelegationDomainService.cs:333`、`PermissionRequestDomainService.cs:536`、`UserDomainService.cs`（平台管理员成员相关）、`TenantDomainService.cs:284`。
  - 抽公共 helper `EnsurePlatformOpsForGlobal(entity)` 收敛重复抛错文案。
- **REQ-1.3** 删除 `SysMenu.cs:36` 误导注释，改为：`TenantId=0 为平台全局菜单，所有租户共享读取（Model A），租户可叠加 TenantId>0 的私有菜单；全局菜单仅平台运维态可维护。`

示例（菜单）：
```csharp
// MenuDomainService 构造注入 ICurrentTenant _currentTenant
private async Task<SysMenu> GetEditableMenuOrThrowAsync(long id, CancellationToken ct)
{
    if (id <= 0) throw new ArgumentOutOfRangeException(nameof(id), "菜单主键必须大于 0。");
    var menu = await _menuRepository.GetByIdAsync(id, ct) ?? throw new InvalidOperationException("菜单不存在。");
    if (menu.IsGlobal && _currentTenant.Id is not null)   // 仅平台态(Id==null)放行
        throw new InvalidOperationException("平台级全局菜单仅平台运维态可维护。");
    return menu;
}
```

### 前端
- 无（本 Phase 纯后端）。

### 验收
- 平台态（无租户 token）可编辑全局菜单/权限/角色等；租户态编辑全局记录仍被拦；租户态编辑自己私有记录不受影响。

## 7. Phase 2 租户切换与平台态入口

**目标**：让用户在多租户间切换、让平台超管进入平台态——这是 Phase 1 生效的前提（你的需求 2）。

### 后端
- **REQ-2.1** 新增切换接口 `POST /Auth/SwitchTenant`。
  - 入参：`{ tenantId?: long }`（null/0 = 切到平台态）。
  - 逻辑：
    1. 取当前用户 `ICurrentUser.UserId`；
    2. 目标为某租户：校验 `SysTenantUser` 成员有效（`InviteStatus=Accepted && Status=Valid`）；super_admin 可切任意租户；
    3. 目标为平台态：仅 `IsSystemAccount`/super_admin/`PlatformAdmin` 成员允许；
    4. 重新计算该上下文下的角色/权限快照，调用 `AuthTokenIssueService.IssueAccessToken` 重签发（目标租户→带 TenantId claim；平台态→不带）；
    5. 返回新的 `LoginTokenDto`。
  - 文件：`Application/AppServices/Auth/AuthAppService.cs`(+方法)、`Application/Contracts/.../IAuthAppService.cs`、`Application/Dtos/Auth/AuthDtos.cs`(+`SwitchTenantRequestDto`)；复用 `AuthTokenIssueService`、`AuthorizationSnapshotQueryService`、`TenantUserRepository`。
- **REQ-2.2** 平台态会话标识：在快照/`UserInfo` 返回 `isPlatform: bool`、`currentTenantId?`、`memberType`，供前端渲染。
  - 文件：`QueryServices/Auth/*`、`Dtos/Auth/AuthDtos.cs`。
- **REQ-2.3**（可选）登录前按租户编码解析：`GetLoginTenantOrThrowAsync` 已支持 id；如需支持 code，复用 `ITenantStore.FindAsync(string)`。

### 前端
- **REQ-2.4** API：`frontend/src/app/context.ts` 增 `switchTenantApi(tenantId?)` → `POST /Auth/SwitchTenant`，成功后 `afterLogin(token)` 刷新会话。
- **REQ-2.5** 登录前选择：`frontend/packages/views/_core/authentication/login.vue` 当 `tenantEnabled` 时把 `tenantId` 输入升级为可选列表/记忆上次；保留留空=平台态（受后端校验约束）。
- **REQ-2.6** 登录后切换器：`frontend/packages/views/_core/profile/ProfileTabTenants.vue` 接 `switchTenantApi`；顶栏增加租户切换下拉（展示当前租户/平台态）。
- **REQ-2.7** 平台运维入口：当 `isPlatform || 可切平台` 时，在用户菜单/顶栏显示「进入平台运维」（切平台态）与「返回租户」。
  - 文件：布局/顶栏组件（`frontend/packages/views/_core/*` 或 `frontend/src/layouts/*`）、`frontend/packages/stores/auth.ts`（持久化当前上下文）。
- **REQ-2.8** 切换后刷新：路由守卫/权限 store 重新拉取 `Permissions`/菜单树。

### API 契约
```
POST /Auth/SwitchTenant
  Req:  { "tenantId": 2 }        // 省略或 null/0 => 平台态
  Resp: { "accessToken":"...", "refreshToken":"...", "tokenType":"Bearer", "expiresIn":..., ... }
GET  /Auth/UserInfo  (扩展)
  Resp: { ...原有..., "isPlatform": false, "currentTenantId": 2, "memberType": "Owner" }
```

### 验收
- 多租户成员可在租户间切换并看到对应数据；平台超管可切到平台态后编辑全局模板（联动 Phase 1）；非成员切换被拒。

## 16. Phase 7 — 登录身份全局化与控制中心（登录后选租户）

> 状态：📋 已评估 + 已定决策，**待落地**（2026-06-09）

### 16.1 目标流程（产品诉求）

```
登录页：邮箱 + 密码（不选择租户）
   └─ 后端按邮箱全局解析用户 → 校验密码 → 读成员关系(SysTenantUser) → 决定落点：
        ├─ 超管 / 平台账号       → 平台态令牌 → 落「控制中心」(管租户/用户/系统 + 可选某租户进入)
        ├─ 恰好 1 个有效租户     → 直接签发该租户令牌 → 落应用首页（免选）
        └─ 多个有效租户          → 平台态令牌 → 落「控制中心」选租户 → SwitchTenant 进入
进入租户后仍可随时切换租户 / 超管可切回平台态。
```

### 16.2 可行性结论

**能实现，但当前架构是"反方向"设计**——`SysUser.cs:41-42` 明确为**两段式登录**：先选租户定位 `TenantId`，再 `WHERE TenantId + UserName` 唯一定位（`UX_TeId_UsNa`，用户名仅租户内唯一）；登录解析见 `SaasUserStore.cs:51-54`（`UserName + (CurrentTenant.Id ?? 0)`）。因此"先登录、后选租户"需要一次**登录身份模型调整**：把登录标识从「按租户唯一」改为「全局唯一」。

运行时骨架已由 Phase 1-2 具备，可直接复用：
- 平台态上下文（框架 filter + token 支持 `Id=null`）。
- 跨租户成员查询 `ITenantUserRepository.GetActiveByUserIdAsync`（一人多租户）。
- 切换租户重签发 `POST /Auth/SwitchTenant`（成员校验 + 平台态放行）。
- 平台态可维护全局模板（Phase 1）。

### 16.3 决策（已确认）

| # | 决策 | 取值 |
|---|---|---|
| ⑥ 登录标识 | 全局唯一键 | **邮箱全局唯一**（用户名仍可租户内重复） |
| ⑦ 落点策略 | 登录后路由 | 超管/平台账号→控制中心；普通用户**单租户直进、多租户选择**；零租户→控制中心/提示 |
